## 1. TanStack Start が React Server Components を再考する

**記事タイトル**: React Server Components Your Way
**URL**: [https://tanstack.com/blog/react-server-components](https://tanstack.com/blog/react-server-components)

### ① 目的・問題意識

Next.js をはじめとする既存の RSC 実装はサーバーがコンポーネントツリーを所有する「サーバーファースト」モデルを採用している。これにより次のような問題が生じていた。

- 開発者が既存の SPA メンタルモデルと摩擦を覚える
- RSC のキャッシュ戦略が TanStack Query など既存のクライアントキャッシュと相容れない独自モデルを要求する
- `'use server'` ディレクティブに起因する重大なセキュリティ脆弱性（CVE-2025-55182 / React2Shell）が明らかになった

TanStack Start はこれらの課題に対し、**クライアントファースト・RSC オプトイン**という真逆のアプローチを採る。

### ② 具体的な技術詳細とコード例

#### 基本的な RSC の定義と返却

```typescript
// app/server/greeting.tsx
import { createServerFn } from '@tanstack/react-start'
import { renderToReadableStream } from '@tanstack/react-start/rsc'
import { setResponseHeaders } from '@tanstack/react-start/server'

// RSC ストリームをサーバー関数として返す
const getGreeting = createServerFn({ method: 'GET' }).handler(async () => {
  setResponseHeaders(
    new Headers({
      'Cache-Control': 'public, max-age=0, must-revalidate',
      'Netlify-CDN-Cache-Control':
        'public, max-age=300, durable, stale-while-revalidate=300',
    }),
  )
  // サーバーコンポーネントをストリームとして返す
  return renderToReadableStream(<h1>Hello from the server</h1>)
})
```

#### ルートローダーと RSC の統合

```typescript
// app/routes/index.tsx
import { createFileRoute } from '@tanstack/react-router'
import { createServerFn } from '@tanstack/react-start'
import { renderServerComponent } from '@tanstack/react-start/rsc'

// サーバー側でのみ実行されるコンポーネント
function Greeting() {
  return <h1>Hello from RSC</h1>
}

// RSC をサーバー関数でラップしてクライアントに渡す
const getGreeting = createServerFn().handler(async () => {
  const Renderable = await renderServerComponent(<Greeting />)
  return { Renderable }
})

export const Route = createFileRoute('/')({
  // ルートローダーで RSC を取得（TanStack Query と同一キャッシュで管理可能）
  loader: async () => {
    const { Renderable } = await getGreeting()
    return { Greeting: Renderable }
  },
  component: HomePage,
})

function HomePage() {
  const { Greeting } = Route.useLoaderData()
  return <>{Greeting}</>
}
```

#### `'use server'` を使わない理由（セキュリティ）

```typescript
// TanStack Start は 'use server' ディレクティブを使用しない。
// 代わりに createServerFn で明示的な RPC エンドポイントを定義する。
// これにより Flight プロトコルの汎用エンドポイントが存在せず、
// React2Shell (CVE-2025-55182) などの脆弱性が適用されない。

// ❌ Next.js / 従来型 RSC の 'use server' — CVE 対象
// async function myAction(data: FormData) {
//   'use server'
//   await db.insert(data)
// }

// ✅ TanStack Start の明示的 RPC
const myAction = createServerFn({ method: 'POST' })
  .validator((data: FormData) => data) // 入力検証を明示
  .handler(async ({ data }) => {
    await db.insert(data)
  })
```

#### RSC と TanStack Query の統合

```typescript
import { useQuery } from '@tanstack/react-query'

function MyComponent() {
  // RSC ペイロードを通常のデータクエリと同一のキャッシュ機構で管理できる
  const { data: Greeting } = useQuery({
    queryKey: ['greeting'],
    queryFn: () => getGreeting().then(r => r.Renderable),
    staleTime: 5 * 60 * 1000, // 5 分キャッシュ
  })

  return <div>{Greeting}</div>
}
```

#### パフォーマンス実績

TanStack 公式サイトのコンテンツ重視ページで RSC 移行後に計測した結果:

| 指標 | 移行前 | 移行後 |
|------|--------|--------|
| クライアントバンドルサイズ（gzip） | ベースライン | **〜153 KB 削減** |
| Total Blocking Time（ブログ記事ページ） | 1,200 ms | **260 ms** |

> ダッシュボードや操作が多いページではほぼ改善なし — RSC は全ページに適用するものではなく、コンテンツ重視ページで選択的に使うことが推奨される。

### ③ なぜ効果的か

| 観点 | 内容 |
|------|------|
| **クライアントファースト** | SPA のメンタルモデルを壊さず、RSC はオプション追加として機能する |
| **キャッシュの一元管理** | RSC ペイロードが HTTP ストリームとして提供されるため、TanStack Query の `staleTime` や `refetchOnWindowFocus` がそのまま使える |
| **セキュリティ** | Flight プロトコルの汎用エンドポイントを持たないため CVE-2025-55182 等の影響を受けない |
| **段階的採用** | 既存のクライアントルートはそのままに、コンテンツ重視ページだけ RSC を有効化できる |

## 2. Animate UI — 新世代のアニメーション付き React コンポーネント集

**URL**: [https://animate-ui.com](https://animate-ui.com) / [GitHub](https://github.com/imskyleen/animate-ui)

### ① 目的・問題意識

shadcn/ui は高品質な UI コンポーネントを提供するが、アニメーションは最小限に留まる。一方、Motion（旧 Framer Motion）などのアニメーションライブラリを自前で組み込むには学習コストと実装工数がかかる。

Animate UI は **shadcn CLI と同じインターフェース**でアニメーション付きコンポーネントをプロジェクトに取り込める「アニメーション専門のコンポーネントディストリビューション」として設計された。

### ② 具体的な技術詳細とコード例

#### インストール（shadcn CLI 経由）

```bash
# shadcn/ui プロジェクトに Animate UI コンポーネントを追加
npx shadcn@latest add "https://animate-ui.com/r/typewriter"
npx shadcn@latest add "https://animate-ui.com/r/fade-in"
npx shadcn@latest add "https://animate-ui.com/r/slide-in"
```

#### コンポーネント使用例

```tsx
// タイプライターアニメーション
import { Typewriter } from '@/components/animate-ui/typewriter'

export function Demo() {
  return (
    <Typewriter
      text="Hello, World!"
      speed={50}        // ms/文字
      cursor={true}
      cursorChar="|"
    />
  )
}
```

```tsx
// スクロール連動フェードイン
import { FadeIn } from '@/components/animate-ui/fade-in'

export function Section() {
  return (
    <FadeIn
      delay={0.2}
      duration={0.6}
      once={true}        // 一度だけアニメーション
    >
      <h2>Visible on scroll</h2>
      <p>Content fades in when entering viewport.</p>
    </FadeIn>
  )
}
```

#### 技術スタック

| 要素 | 採用技術 |
|------|---------|
| ランタイム | React + TypeScript |
| スタイリング | Tailwind CSS |
| アニメーションエンジン | Motion（旧 Framer Motion） |
| インストール手段 | shadcn CLI（レジストリ形式） |
| ライセンス | MIT（オープンソース） |

#### コンポーネントカテゴリ

- **Animated Primitives** — Button、Badge、Card などの基本要素
- **Text Effects** — Typewriter、CountUp、GradientText
- **Background** — ParticleField、GridBackground、GradientBlob
- **Layout** — FadeIn、SlideIn、StaggerChildren

### ③ なぜ効果的か

- **コピーファースト**: コンポーネントのソースコードが直接プロジェクトに入るため、デザインシステムに合わせた改変が自由
- **shadcn エコシステムとの親和性**: CLI コマンドが共通で学習コストが低い
- **Animation-first かつアクセシブル**: `prefers-reduced-motion` を考慮したデフォルト設定
- **スタイルへの非干渉**: Tailwind のユーティリティクラスで上書き可能な最小限のベーススタイル

## 3. JSON Alexander — Wes Bos 作の JSON ビューアブラウザ拡張

**URL**: [https://github.com/wesbos/JSON-Alexander](https://github.com/wesbos/JSON-Alexander)

### ① 目的・問題意識

API 開発・デバッグ時、ブラウザで JSON エンドポイントにアクセスすると生のテキストが表示される。標準の Chrome DevTools では折りたたみや検索が難しく、JSON パスのコピーも手作業が必要だった。

JSON Alexander は **ブラウザが JSON を返すページを自動検出**し、視覚的な対話型ビューアに置き換えるブラウザ拡張機能。

### ② 具体的な技術詳細

#### 主要機能

| 機能 | 内容 |
|------|------|
| **自動検出** | Content-Type が `application/json` のページで自動的に起動 |
| **3 つの表示モード** | Tree（対話型ツリー）/ Formatted（整形済み）/ Raw（生 JSON） |
| **レベル制御** | 1 / 2 / 3 / All ボタンで展開深度を一括制御 |
| **JSON パス** | プロパティホバーでパスを表示、クリックでピン留め＆クリップボードコピー |
| **コンソール連携** | `window.data` として JSON を DevTools コンソールから操作可能 |
| **テーマ** | Light / Dark / Auto（システム設定連動） |
| **構文ハイライト** | key / string / number / boolean / null をそれぞれ色分け |
| **インデントガイド** | ネスト構造を視覚的に把握するためのガイドライン |

#### インストール

```
Chrome: Chrome ウェブストアから直接インストール
Firefox: マニフェストファイルを手動読み込み
         ※事前に about:config で devtools.jsonview.enabled を false に設定
```

#### 技術スタック

```
TypeScript  73.6%
CSS         26.4%
```

#### 開発コマンド

```bash
# 開発ビルド（ホットリロード）
npm run dev

# 本番ビルド
npm run build

# 配布用パッケージ生成
npm run dist
```

#### 使用例（DevTools コンソール）

```javascript
// JSON Alexander が window.data に JSON を格納するため、
// DevTools コンソールからそのまま操作できる
window.data.users.filter(u => u.role === 'admin')
// → [{id: 1, name: "Alice", role: "admin"}, ...]

// パスのコピー例（ホバーで表示されるパス）
// "users[0].address.city"
```

### ③ なぜ効果的か

- **ゼロ設定**: 拡張インストールのみ、API 側の変更不要
- **`window.data` 連携**: ブラウザコンソールから直接 JSON を操作でき、フィルタリング・デバッグが素早い
- **パスコピー**: フロントエンド実装時に必要な JSON パスをワンクリックでコピー

## 4. 「過剰だった」と気づいた React パターン集

**参考記事**: "Why Most React Apps Are Overengineered" (Medium, Apr 2026) 他

### ① 目的・問題意識

React Compiler v1.0（2025年10月安定版）の登場と React 19 の新機能により、これまで「ベストプラクティス」とされていたパターンの多くが**不要な複雑さ**に変わった。特に AI コード生成が普及した 2026 年には「どのパターンを実装するか」より「**どのパターンを使わないか**」が上級者の証とされている。

### ② 過剰とみなされるパターンと代替案

#### パターン 1: 手動 `useMemo` / `useCallback`

```tsx
// ❌ 2023年以前の「最適化」— React Compiler が自動で行う
function ProductList({ products }: { products: Product[] }) {
  const sortedProducts = useMemo(
    () => products.sort((a, b) => a.price - b.price),
    [products]
  )
  const handleClick = useCallback(
    (id: string) => router.push(`/products/${id}`),
    [router]
  )
  return <ul>{sortedProducts.map(p => (
    <li key={p.id} onClick={() => handleClick(p.id)}>{p.name}</li>
  ))}</ul>
}

// ✅ 2026年 — React Compiler がビルド時に自動メモ化
function ProductList({ products }: { products: Product[] }) {
  const sortedProducts = products.sort((a, b) => a.price - b.price)
  return <ul>{sortedProducts.map(p => (
    <li key={p.id} onClick={() => router.push(`/products/${p.id}`)}>{p.name}</li>
  ))}</ul>
}
```

#### パターン 2: 過度な状態管理ライブラリの導入

```tsx
// ❌ サイドバーの開閉を管理するだけに Redux + Context + Zustand を同居させる例
// （実際のコードベースで見られた過剰設計）
const useSidebarStore = create<SidebarState>(set => ({
  isOpen: false,
  toggle: () => set(state => ({ isOpen: !state.isOpen })),
}))
const SidebarContext = createContext<SidebarContextType | null>(null)
// さらに Redux slice も存在...

// ✅ useState で十分
function App() {
  const [isSidebarOpen, setIsSidebarOpen] = useState(false)
  return (
    <>
      <Sidebar isOpen={isSidebarOpen} />
      <button onClick={() => setIsSidebarOpen(v => !v)}>Toggle</button>
    </>
  )
}
```

#### パターン 3: `useEffect` でのデータ取得

```tsx
// ❌ useEffect + 手動ローディング/エラー状態管理
function UserProfile({ userId }: { userId: string }) {
  const [user, setUser] = useState<User | null>(null)
  const [loading, setLoading] = useState(true)
  const [error, setError] = useState<Error | null>(null)

  useEffect(() => {
    setLoading(true)
    fetchUser(userId)
      .then(setUser)
      .catch(setError)
      .finally(() => setLoading(false))
  }, [userId])

  if (loading) return <Spinner />
  if (error) return <ErrorMessage error={error} />
  return <div>{user?.name}</div>
}

// ✅ TanStack Query（または React 19 の use()）
function UserProfile({ userId }: { userId: string }) {
  const { data: user, isLoading, error } = useQuery({
    queryKey: ['user', userId],
    queryFn: () => fetchUser(userId),
  })

  if (isLoading) return <Spinner />
  if (error) return <ErrorMessage error={error} />
  return <div>{user?.name}</div>
}
```

#### パターン 4: コンポーネント内コンポーネント定義

```tsx
// ❌ 再レンダリングのたびに新しいコンポーネントが生成される
function ParentComponent() {
  // InnerComponent はレンダリングごとに新しい参照になりパフォーマンスが落ちる
  function InnerComponent({ text }: { text: string }) {
    return <p>{text}</p>
  }
  return <InnerComponent text="hello" />
}

// ✅ モジュールスコープで定義
function InnerComponent({ text }: { text: string }) {
  return <p>{text}</p>
}
function ParentComponent() {
  return <InnerComponent text="hello" />
}
```

#### パターン 5: `useEffect` での派生状態管理

```tsx
// ❌ 派生状態を useEffect で同期する（アンチパターン）
function SearchResults({ items, query }: { items: Item[]; query: string }) {
  const [filtered, setFiltered] = useState(items)

  useEffect(() => {
    setFiltered(items.filter(i => i.name.includes(query)))
  }, [items, query])

  return <ul>{filtered.map(i => <li key={i.id}>{i.name}</li>)}</ul>
}

// ✅ レンダリング中に直接計算
function SearchResults({ items, query }: { items: Item[]; query: string }) {
  const filtered = items.filter(i => i.name.includes(query)) // 派生状態は計算で
  return <ul>{filtered.map(i => <li key={i.id}>{i.name}</li>)}</ul>
}
```

### ③ なぜ効果的か

- **React Compiler の自動最適化**: 手動 `useMemo`/`useCallback` はほぼ不要になり、コードが読みやすくなる
- **AI コード生成時代の「シンプルさ」**: 生成コストが下がるほど、設計のシンプルさが競争優位に直結する
- **保守コスト**: フォームフィールド追加が 5 ファイル更新を要求するアーキテクチャは財務的なリスク
