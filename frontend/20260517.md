## 1. CSS `random()` 関数 — CSS でネイティブにランダム値を生成する

**記事タイトル**: Very Early Playing with random() in CSS / Rolling the Dice with CSS random()
**主要 URL**:
- [Frontend Masters Blog](https://frontendmasters.com/blog/very-early-playing-with-random-in-css/)
- [WebKit Blog](https://webkit.org/blog/17285/rolling-the-dice-with-css-random/)
- [MDN Web Docs](https://developer.mozilla.org/en-US/docs/Web/CSS/Reference/Values/random)
- [CSS-Tricks Almanac](https://css-tricks.com/almanac/functions/r/random/)

### ① 目的・問題意識

CSS でランダムな値を実現するにはこれまで以下の手段が必要だった。

- **Sass/PostCSS などプリプロセッサ** のビルド時ループで変数を埋め込む（ビルド後は静的なため「見かけのランダム」）
- **JavaScript で `style.setProperty()` を使い** インラインカスタムプロパティを動的に設定する
- **`:nth-child` ハック** — 要素ごとに異なる固定値を列挙する

これらはいずれもメンテナンスコストが高く、真の意味での「レンダリングタイム・ランダム性」ではない。CSS Values and Units Module Level 5 仕様で策定された `random()` 関数は、**スタイルシート内で直接ランダムな数値を生成できる** ネイティブ機能を提供する。

### ② 具体的なコードと技術詳細

#### 基本構文

```css
/* 基本形: MIN と MAX を指定 */
random(MIN, MAX)

/* ステップ付き */
random(MIN, MAX, step STEP)

/* シード付き（値の共有制御） */
random(SEED, MIN, MAX)
random(SEED, MIN, MAX, step STEP)
```

- **MIN / MAX** — 同一の単位型（length・angle・time など）を使う必要がある。`rem` と `px` の混在は不可
- **STEP** — 戻り値を MIN から始まる STEP 刻みに制限する（`step 10px` なら 10px・20px・30px …）
- **SEED** — `random-value-sharing` パラメータ。値の「共有範囲」を制御する

#### SEED の種類

| SEED 値 | 挙動 |
|---------|------|
| 省略 | ルールレベルで一度評価（同じルールにマッチする全要素が同じ値を受け取る） |
| `per-element` | 要素ごとに個別評価（各要素が異なるランダム値を受け取る） |
| `--custom-ident` | 同一要素内で同じダッシュ識別子を持つ `random()` 呼び出しが同じ値を共有する |

#### 使用例

```css
/* 各カードを異なる位置にランダム配置 */
.card {
  top: random(per-element, 5rem, 20rem);
  left: random(per-element, 0rem, 30rem);
}

/* ランダムな回転角 */
.sticker {
  rotate: random(per-element, -15deg, 15deg);
}

/* アニメーションディレイを要素ごとにバラバラに */
.particle {
  animation-delay: random(per-element, 0s, 3s);
}

/* 幅と高さを同じランダム値にする（正方形）: --size ident を共有 */
.box {
  width: random(--size, 100px, 200px);
  height: random(--size, 100px, 200px);
}

/* ステップ付き: 50px 刻みでランダム */
.grid-item {
  height: random(per-element, 100px, 400px, step 50px);
}
```

#### プログレッシブエンハンスメント（@supports）

```css
/* フォールバック値を先に定義 */
.card {
  top: 10rem;
}

/* random() がサポートされているブラウザでのみ適用 */
@supports (top: random(1px, 2px)) {
  .card {
    top: random(per-element, 5rem, 20rem);
  }
}
```

#### ブラウザサポート状況（2026年5月時点）

| ブラウザ | サポート |
|---------|---------|
| Safari 26.2+ | ✅ 安定版でサポート（`element-scoped` は Safari 26.5 で追加） |
| Chrome/Edge | 🔄 仕様追跡中、実装予定 |
| Firefox | 🔄 Positive Standards Position、追跡中 |

### ③ なぜそれが効果的か

CSS だけでランダムなレイアウト・アニメーションを実現できる意義：

- **JavaScript 不要** — DOM 操作やイベントリスナーなしにランダム性を付与できる
- **CSS の宣言的モデルと整合** — スタイルと振る舞いが CSS 内で完結し、コンポーネントの独立性が高まる
- **パフォーマンス** — JS によるレイアウト計算を排除し、ブラウザ最適化の対象となる
- **将来の cross-browser 化が見込まれる** — Firefox・Chrome も Positive Position を示しており、Interop ターゲットになり得る

## 2. ウェブプラットフォームがライブラリから影響を受けた 9 つの事例

**記事タイトル**: 9 Times the Web Platform Was Influenced by Libraries
**著者**: Jad Joubran
**URL**: [https://jadjoubran.io/blog/web-platform-influenced-by-libraries](https://jadjoubran.io/blog/web-platform-influenced-by-libraries)

### ① 目的・問題意識

「この API、どこかで見たことがある」という感覚の正体を探る記事。Web 標準は真空の中で生まれるのではなく、**先に普及したライブラリが示したパターンを後から仕様が追認する** という形でその多くが生まれてきた。querySelector から Temporal まで、9 つの具体例を通じてその「ライブラリ → プラットフォーム」サイクルを解説する。

### ② 具体的なコードと技術詳細

#### 例 1: Promises（非同期処理）

```javascript
// Before: ライブラリ時代（jQuery Deferred / Q / Bluebird）
$.ajax({ url: '/api' })
  .done(data => console.log(data))
  .fail(err => console.error(err));

// After: ネイティブ Promise
fetch('/api')
  .then(res => res.json())
  .then(data => console.log(data))
  .catch(err => console.error(err));
```

jQuery Deferred・Q・Bluebird が独自の非同期モデルを競い合い、最終的にコミュニティが Promises/A+ 仕様を策定、それを受けて ECMAScript が `Promise` を組み込んだ。

#### 例 2: querySelector（DOM セレクタ）

```javascript
// Before: jQuery の $ セレクタ
$('.card.active')
$('#header a')

// After: ネイティブ DOM API
document.querySelector('.card.active')
document.querySelectorAll('#header a')
```

jQuery の CSS セレクタによる要素取得が普及し、`querySelector` / `querySelectorAll` が標準化された。

#### 例 3: structuredClone（ディープクローン）

```javascript
// Before: Lodash の cloneDeep
import { cloneDeep } from 'lodash';
const copy = cloneDeep(original);

// After: ネイティブ structuredClone
const copy = structuredClone(original);
// Date、Map、Set、RegExp、TypedArray も正しくクローンされる
```

`structuredClone()` はブラウザが内部的に長年使ってきたアルゴリズムを公開 API として露出させたもの。

#### 例 4: Fetch API（HTTPリクエスト）

```javascript
// Before: XMLHttpRequest（verbose）
const xhr = new XMLHttpRequest();
xhr.open('GET', '/api/data');
xhr.onload = () => { /* ... */ };
xhr.send();

// After: Fetch API + async/await
const data = await fetch('/api/data').then(r => r.json());
```

axios・superagent などのラッパーが示した使いやすいインターフェースが Fetch API として標準化された。

#### 例 5: Temporal API（日付・時刻処理）

```javascript
// Before: Moment.js
const dt = moment('2026-05-06').add(7, 'days').format('YYYY-MM-DD');

// After: Temporal API (TC39 Stage 3 → Stage 4)
const dt = Temporal.PlainDate.from('2026-05-06')
  .add({ days: 7 })
  .toString(); // "2026-05-13"
```

Moment.js・date-fns・Luxon が積み上げた「不変（immutable）・タイムゾーン対応」のベストプラクティスが Temporal API の設計原則になった。

#### 例 6: Container Queries（コンテナクエリ）

```css
/* Before: Element Queries ライブラリ（JS ベース） */
/* [data-eq~="min-width:300px"] .card { ... } */

/* After: ネイティブ CSS Container Queries */
.widget {
  container-type: inline-size;
}

@container (min-width: 300px) {
  .card { flex-direction: row; }
}
```

Phil Hawksworth らが提唱した "Element Queries" の概念が CSS Container Queries として標準化。

#### その他の例（7〜9）

- **`Array` メソッド群**（`map`・`filter`・`reduce`）: Underscore.js・Lodash が提示したコレクション操作が ECMAScript に組み込まれた
- **`Element.closest()`**: jQuery の `.closest()` が同名の DOM API として標準化
- **CSS カスタムプロパティ（CSS 変数）**: Sass/Less 変数のエコシステムが CSS Custom Properties `--var` として仕様化

### ③ なぜそれが効果的か

- **「ライブラリ不要論」を根拠を持って語れる** — 「何故今はネイティブで OK なのか」を歴史的文脈で説明できる
- **仕様策定プロセスの理解** — WHATWG/W3C/TC39 は机上で設計するのではなく、実際のエコシステムからフィードバックを受けて標準化することがわかる
- **未来の仕様を予測するヒント** — 現在広く使われているライブラリのパターンが将来の Web 標準になるかもしれない

## 3. Chrome 148 の新機能 — video/audio の lazy loading・Prompt API など

**記事タイトル**: New in Chrome 148
**URL**: [https://developer.chrome.com/blog/new-in-chrome-148](https://developer.chrome.com/blog/new-in-chrome-148)
**リリース日**: 2026年5月5日（stable）

### ① 目的・問題意識

Chrome 148 では複数の重要な web platform 機能が同時にリリースされた。特に注目されるのは：

1. `<video>`・`<audio>` 要素の **native lazy loading** — 長年の実装ギャップを埋める
2. **CSS name-only container queries** — コンテナタイプ不要でクエリ可能に
3. **Prompt API** — ブラウザ組み込み AI (Gemini Nano) を Web アプリから直接利用
4. **`at-rule()` in `@supports`** — CSS アットルールのサポートチェック
5. **`revert-rule`** — 特定のルールのみをリバート

### ② 具体的なコードと技術詳細

#### 機能 1: `<video>` / `<audio>` の `loading="lazy"`

```html
<!-- ビューポートに近づくまで読み込みを遅延 -->
<video loading="lazy" src="large-video.mp4" controls></video>
<audio loading="lazy" src="podcast.mp3" controls></audio>

<!-- poster 画像・autoplay もビューポート近接まで遅延 -->
<video loading="lazy" src="video.mp4" poster="thumbnail.jpg" autoplay muted></video>
```

**効果**: `loading="lazy"` を付加するだけで、要素がビューポートに近づくまでデータ取得・autoplay・poster 読み込みを延期。JavaScript 不要。未対応ブラウザは属性を無視するため後方互換性あり。

#### 機能 2: CSS Name-only Container Queries

```css
/* Before: container-type が必須だった */
.sidebar {
  container-name: sidebar;
  container-type: inline-size; /* 必須だった */
}

/* After: container-name だけでクエリ可能 */
.sidebar {
  container-name: sidebar;
  /* container-type 省略可能 */
}

@container sidebar (min-width: 300px) {
  .widget { flex-direction: row; }
}
```

`container-type` を省略した「名前のみコンテナ」でスタイルクエリを使用できるようになった。特定コンテナを名指しする場合に余分なサイズ制約を課す必要がなくなる。

#### 機能 3: Prompt API（Gemini Nano を Web アプリから利用）

```javascript
// オンデバイス AI セッションを生成
const session = await ai.languageModel.create({
  systemPrompt: 'あなたは丁寧なカスタマーサポートエージェントです。'
});

// テキスト生成
const response = await session.prompt('注文のキャンセル方法を教えてください');
console.log(response);

// ストリーミング生成
const stream = session.promptStreaming('製品の特徴をまとめてください');
for await (const chunk of stream) {
  outputElement.textContent += chunk;
}
```

Prompt API は Chrome 組み込みの **Gemini Nano** を通じてオンデバイス推論を提供。インターネット接続なしでも動作し、サーバーコストなしでテキスト生成・分類・要約などを Web アプリに統合できる。テキスト・画像・音声入力をサポート。

#### 機能 4: `@supports at-rule()`

```css
/* CSS アットルールのサポートを確認 */
@supports at-rule(@container) {
  .widget {
    container-type: inline-size;
  }
}

@supports at-rule(@layer) {
  @layer utilities {
    .text-center { text-align: center; }
  }
}
```

#### 機能 5: `revert-rule`

```css
.button {
  background: blue;
  color: white;
  padding: 8px 16px;
}

/* 特定プロパティのみをルール前の状態に戻す */
.button.ghost {
  background: revert-rule;  /* .button の background: blue を取り消す */
  border: 2px solid blue;
}
```

`revert-rule` は `revert` と異なり、**現在のルールが適用する前**（つまり前のルール）に値を戻す。カスケードの精緻な制御が可能になる。

### ③ なぜそれが効果的か

- **`loading="lazy"` for media**: `<img>` では既にサポートされていた lazy loading が `<video>`・`<audio>` に拡張されることで、メディアリッチなページのパフォーマンスが JS なしで改善できる
- **Prompt API**: クラウドに依存しないオンデバイス推論で、プライバシーを保ちながら AI 機能を Web アプリに組み込める
- **Name-only container queries**: コンテナのサイジングコンテキストを変更せずにスタイルクエリが使えるので、既存レイアウトへの影響が少ない

## 4. CSS Media Query Range Syntax — 比較演算子で書くメディアクエリ

**記事タイトル**: Media Queries Range Syntax
**URL**: [https://ishadeed.com/article/range-syntax/](https://ishadeed.com/article/range-syntax/)
**参考**: [web.dev — New syntax for range media queries](https://web.dev/articles/media-query-range-syntax)

### ① 目的・問題意識

従来の `min-width` / `max-width` 構文は以下の問題を抱えていた。

- 「以上」「以下」「未満」「超」の区別が直感的でない（`min-width: 300px` は 300px **を含む**）
- 境界値で両方のクエリが一致し、バグが起きやすい（`max-width: 300px` と `min-width: 300px` が同時にマッチ）
- 範囲を表現するときに `and` で冗長につなぐ必要がある

Media Queries Level 4 で策定された **Range Syntax** は数学的な比較演算子を使い、より読みやすく・デバッグしやすいクエリを提供する。

### ② 具体的なコードと技術詳細

#### 基本的な比較演算子

```css
/* 従来の書き方 */
@media (min-width: 600px) { /* ... */ }
@media (max-width: 599px) { /* ... */ }

/* Range Syntax */
@media (width >= 600px) { /* ... */ }
@media (width < 600px)  { /* ... */ }
```

#### 範囲指定

```css
/* 従来: and で連結 */
@media (min-width: 30em) and (max-width: 50em) {
  .layout { max-width: 800px; }
}

/* Range Syntax: 数学的な範囲表記 */
@media (30em <= width <= 50em) {
  .layout { max-width: 800px; }
}
```

#### 利用可能な演算子

| 演算子 | 意味 |
|--------|------|
| `<`    | 未満（境界値を含まない） |
| `<=`   | 以下（境界値を含む） |
| `>`    | 超（境界値を含まない） |
| `>=`   | 以上（境界値を含む） |

#### Container Queries への適用

```css
.sidebar {
  container-type: inline-size;
  container-name: sidebar;
}

/* コンテナクエリでも Range Syntax が使える */
@container sidebar (300px <= inline-size <= 600px) {
  .widget {
    flex-direction: row;
  }
}

@container sidebar (inline-size >= 600px) {
  .widget {
    grid-template-columns: 1fr 1fr;
  }
}
```

#### 実践的な使用例（レスポンシブレイアウト）

```css
/* モバイル優先のブレイクポイント設計 */

/* スマートフォン（〜599px） */
.card { flex-direction: column; }

/* タブレット（600px〜899px） */
@media (600px <= width < 900px) {
  .card { flex-direction: row; }
}

/* デスクトップ（900px〜） */
@media (width >= 900px) {
  .card {
    flex-direction: row;
    max-width: 1200px;
  }
}
```

| ブラウザ | サポートバージョン |
|---------|-----------------|
| Chrome  | 104+（90%+ の利用シェア）|
| Firefox | 63+              |
| Safari  | 16.4+            |
| Edge    | 104+             |

グローバルサポート率 90% 以上で、**本番利用が安全**な状態になっている。

### ③ なぜそれが効果的か

- **可読性の向上**: `min-width: 600px` より `width >= 600px` の方が意図が明確
- **境界値バグの防止**: `<`（未満）と `<=`（以下）を明確に使い分けることで、ブレイクポイント境界での意図しない両適用を防げる
- **コードの対称性**: 「以上〜以下」を一行で表現でき、ロジックが追いやすくなる
- **CSS Container Queries にも同じ構文が使える**: 学習コストなしで適用範囲を拡大できる
