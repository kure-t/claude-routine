## 1. Projecting React — TanStack の実験的 React クローン (@tanstack/redact)

**記事リンク**: [Projecting React — Tanner Linsley](https://tannerlinsley.com/posts/projecting-react)  
**GitHub**: [TanStack/redact](https://github.com/TanStack/redact)  
**npm**: [@tanstack/redact](https://www.npmjs.com/package/@tanstack/redact)

---

### ① 目的・問題意識：何を解決する技術か

**問題**: React は大きい。Vite 経由でバンドルすると、クライアントにアプリコードを1行も書く前に **~45〜60 KB gzip** が必要になる。TanStack Start の技術スタックの中で「外せない最小ピース」が「最大のコスト」というアンバランスがあった。

**代替案 Preact を検討したが断念**:  
`preact/compat` は「小さな React」として有力候補だったが、React 19 から乖離が大きく、`use()` のセマンティクス・React 19 Server Actions・ポータル・エラーバウンダリ・ハイドレーションの挙動が一致しなかった。

**着想**: コーディングエージェントの台頭によって「コードは Base Table ではなく、アイデア（アルゴリズム・プロトコル・セマンティック契約）が Base Table であり、コードはその Projection の一つに過ぎない」という考え方が成立するようになった。  
React の Public API（要素モデル・フックの契約・Suspense のセマンティクス・ハイドレーションライフサイクル・SSR ストリーミングプロトコル）は10年間で磨かれた安定した Base Table である。React-the-repo はその Projection のひとつに過ぎず、TanStack Start が必要としないユースケース（concurrent mode、time slicing、DevTools など）に最適化されたものだ。

**解決策**: TanStack Start に必要なスコープのみを満たす **別の Projection** を AI エージェントで生成した。これが **`@tanstack/redact`**。

#### Vite プラグインによるドロップイン置換


| プリセット | サイズ | 戦略 |

| `full` | 9.07 KB | 全機能ON、不要なものだけ opt-out |
| `nano` | 6.75 KB | 全機能OFF、必要なものだけ opt-in |

**full プリセットから不要機能を除外する例**:

```typescript
// vite.config.ts
import { redact } from '@tanstack/redact/vite'

export default {
  plugins: [
    redact({
      preset: 'full',
      features: {
        hydration: false,       // SPA のみ (SSR なし) の場合
        classComponents: false, // 関数コンポーネントのみの場合
      },
    }),
  ],
}
```

**nano プリセットから必要機能だけを opt-in する例**:

```typescript
redact({
  preset: 'nano',
  features: {
    context: true,   // Context API を使う場合
    suspense: true,  // Suspense を使う場合
  },
})
```

#### プロジェクト構成

```
packages/redact/src/
  ├── core/          VDOM型・内部シンボル
  ├── react/         createElement、hooks、context
  ├── dom/           Fiber Reconciler、ホスト DOM
  │   └── features/  opt-in機能（各々 index.ts / full.ts / stub.ts）
  ├── server/        SSR レンダリング
  ├── scheduler/     スケジューラシム（最小実装）
  └── vite/          Vite プラグイン
```

#### 実装の技術的な発見

Fiber Reconciler を再実装する過程でリアルトラフィックが明らかにした問題が2つあった:

1. **reconciliation の順序問題**: `placeChildrenInOrder` が順方向で反復すると、子要素の挿入順が狂うたびにカスケード失敗が起きた。**逆順反復**に変えることで解決。
2. **`useEffect` クリーンアップのタイミング**: クリーンアップ実行タイミングは「dispatch 時」ではなく「passive drain（passive effect の実行フェーズ）」にしなければならない。

#### 開発コマンド

```bash
pnpm install
pnpm build       # dist/ 出力 + 型定義生成
pnpm test        # 707 テスト実行
pnpm size        # gzip/brotli サイズレポート
pnpm --filter ssr-demo dev  # SSR デモサーバー起動
```

#### パフォーマンス結果

- **速度**: TanStack Start のワークロードで **2〜3倍高速**
- **Lighthouse**: FCP で改善（特にコンテンツ量の多いドキュメント・ブログページ）
- **TBT / CLS**: ゼロを維持
- **本番稼働**: `tanstack.com` と `tannerlinsley.com` の2サイトで稼働中（既知のリグレッションなし）

---

### ③ なぜ効果的か

1. **「代替フレームワーク」ではなく「同じ API の別 Projection」**: React の Public API を捨てず、その実装だけを最適化。既存コードの書き換えが不要。
2. **AI 生成による低コスト**: 1日のプロンプトエンジニアリングで React サーフェスのほぼ全体（要素モデル・JSX・Fragment・memo・lazy・フルフックセット）が動作するようになった。再生成コストが低いため「特定スコープへの最適化」が現実的な選択肢になった。
3. **段階的に削減可能**: 機能フラグで必要な機能だけを残せるため、SPA・SSR・フル機能と用途に合わせたバンドルサイズを実現できる。
4. **本物の RSC 対応**: Flight シリアライズは実際の `react-server-dom` に委ねているため、RSC 機能を損なわずにクライアント側だけを軽量化できる。

## 2. Next.js / React — 2026年5月 セキュリティリリース (13 CVEs)

**参照**: [Next.js May 2026 security release — Vercel](https://vercel.com/changelog/next-js-may-2026-security-release)  
**参照**: [WAF and framework adapter mitigations — Cloudflare](https://developers.cloudflare.com/changelog/post/2026-05-06-react-nextjs-vulnerabilities/)

---

### ① 目的・問題意識：何を解決する技術か

2026年5月6〜7日、Vercel は Next.js に対する13件の CVE（脆弱性）の修正版をリリースした。加えて、React Server Components (RSC) の upstream に1件の DoS 脆弱性が発見・修正された。前回（2025年12月）の RSC 脆弱性に続く「また別の RSC 脆弱性（another RSC vulnerability）」として React Status のサブタイトルでも言及された。

**重要な注意点**: Vercel が明示しているように、**WAF レイヤーでの緩和は信頼できず**、アップグレードのみが完全な緩和策となる。

---

### ② 具体的な CVE と技術詳細

#### パッチ適用済みバージョン

| パッケージ | 修正バージョン |
|----------|-------------|
| Next.js | **15.5.18** および **16.2.6** |
| react-server-dom-webpack | 19.0.6 / 19.1.7 / **19.2.6** |
| react-server-dom-parcel | 19.0.6 / 19.1.7 / **19.2.6** |
| react-server-dom-turbopack | 19.0.6 / 19.1.7 / **19.2.6** |

#### 脆弱性一覧

**[高] CVE-2026-23870 — React Server Components DoS**
- **対象**: upstream React (react-server-dom-*)
- **概要**: RSC の Flight Protocol を介したサーバー関数ペイロードのデシリアライズ処理に DoS 脆弱性
- **影響**: App Router を使うすべての Next.js デプロイ
- **CVSS**: 7.5 (High)

---

**[緊急] CVE-2026-44578 — WebSocket SSRF**
- **対象**: 自己ホスト型 Next.js (Node.js サーバー)
- **概要**:  
  `Connection: Upgrade` / `Upgrade: websocket` ヘッダを含むリクエストを処理する際、`X-Forwarded-Host` ヘッダの検証が不適切。攻撃者が内部ネットワークや任意の外部ホストへのプロキシトラフィックをサーバーに強制できる。
- **攻撃例**:
  ```
  GET /api/data HTTP/1.1
  Host: target.com
  Connection: Upgrade
  Upgrade: websocket
  X-Forwarded-Host: internal-service.local
  ```
- **影響**: 認証不要でのSSRF。クラウドメタデータサービスへのアクセスや内部サービスの探索が可能。

---

**[高] CVE-2026-44579 — 接続枯渇 DoS (Partial Prerendering)**
- **対象**: Cache Components (Partial Prerendering) を使用する Next.js アプリ
- **概要**: 悪意あるPOSTリクエストがリクエストボディのデッドロックを引き起こし、ファイルディスクリプタが枯渇するまでサーバー接続を開いたままにする。

---

**[高] App Router セグメントプリフェッチ バイパス**
- **概要**: App Router の内部プリフェッチエンドポイントが、ミドルウェアベースの認証チェックを一貫して適用していなかった。
- **影響**: 認証済みエンドポイントへの未認証アクセスが可能になる認証バイパス。

---

**[高] Pages Router i18n デフォルトロケールのプロキシ認可バイパス**
- **概要**: i18n の default-locale パスがプロキシ認可チェックをバイパスする経路を持っていた。

---

**[中] RSC レスポンスのキャッシュポイズニング**
- **概要**: React Server Component のレスポンスがキャッシュポイズニングの対象になる可能性。
- **種類**: RSC レスポンスのキャッシュポイズニング、RSC キャッシュバスティングの衝突によるキャッシュポイズニング（2件）。

---

**[中] XSS — CSP nonce の漏洩**
- **対象**: App Router で CSP nonce を使用するアプリ
- **概要**: `<script nonce={nonce}>` を使った場合に nonce が漏洩し、XSS 攻撃の経路が生じる。

---

**[中] XSS — `beforeInteractive` スクリプト**
- **対象**: `next/script` の `beforeInteractive` を信頼できない入力と組み合わせているアプリ
- **概要**: 入力値がサニタイズされずスクリプトに注入される。

```bash
# または
```

### ③ なぜ効果的（重要）か

- RSC の **Flight Protocol** は強力だが、サーバー側でのペイロードのデシリアライズが攻撃対象になりやすい（今回で2回目の CVE）。
- TanStack Start はサーバー関数に `createServerFn` を使うため、汎用 Flight ペイロードのデシリアライズを行わず、この種の CVE の影響を受けにくい設計になっている。
- 自己ホスト型デプロイ（Vercel 以外）でも適切なパッチ適用が必要。

## 3. TanStack npm サプライチェーン攻撃のポストモーテム

**参照**: [Postmortem: TanStack npm supply-chain compromise — TanStack Blog](https://tanstack.com/blog/npm-supply-chain-compromise-postmortem)  
**参照**: [Hardening TanStack After the npm Compromise — TanStack Blog](https://tanstack.com/blog/incident-followup)

---

### ① 目的・問題意識：何を解決する技術か

2026年5月11日 19:20〜19:26 UTC、攻撃者グループ **TeamPCP**（Mini Shai-Hulud 亜種）が `@tanstack/*` スコープの42パッケージ84バージョンに悪意あるコードを含む npm パッケージを公開した。攻撃は TanStack の正規リリースパイプラインを経由しており、**有効な SLSA Provenance 付き**で公開された初めての悪意ある npm パッケージという点で歴史的な事例となった。

---

### ② 攻撃チェーンの技術詳細

攻撃は3つの脆弱性を組み合わせたもので、それぞれが単独では成立しない。

#### ステップ1: `pull_request_target` Pwn Request

```yaml
# 攻撃に悪用されたワークフローパターン（修正前）
on:
  pull_request_target:  # フォークのPRでもベースリポジトリのコンテキストで実行
    
jobs:
  benchmark-pr:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          ref: ${{ github.event.pull_request.merge_commit_sha }}  # フォークのコードをチェックアウト
```

`pull_request_target` はベースリポジトリのコンテキストで動作するが、フォークの攻撃者制御コードを実行する。これが「Pwn Request」パターン（GitHubセキュリティチームが3年以上前に警告していた既知の危険パターン）。

#### ステップ2: GitHub Actions キャッシュポイズニング

```yaml
# 悪用されたキャッシュキー（修正前）
- uses: actions/cache@v5
  with:
    path: ~/.pnpm-store
    key: Linux-pnpm-store-${{ hashFiles('**/pnpm-lock.yaml') }}
```

- PRワークフロー（`pull_request_target` = ベースリポジトリのキャッシュスコープ）が pnpm-store 内に悪意ある `vite_setup.mjs` を書き込む
- `pnpm-lock.yaml` が変わらなければキャッシュキーが一致するため、次のリリースワークフローが**そのポイズニングされたキャッシュを復元**する

#### ステップ3: OIDC トークンのメモリ抽出

```bash
# 攻撃者が行った操作（概念）
# GitHub Actions ランナーの /proc/<pid>/mem から OIDC トークンを抽出
cat /proc/$(pgrep runner)/mem | grep -o 'ey[A-Za-z0-9._-]*'
```

- `id-token: write` 権限が宣言されているワークフロー内で実行される任意のコードは有効な OIDC トークンを mint できる
- npm の Trusted Publisher バインディングはトークンが `TanStack/router` の `release.yml` から来ていることは検証するが、**ワークフロー内のどのステップが mint したかは検証しない**

#### ペイロードの概要

悪意ある `router_init.js`（2.3 MB、単一行の難読化コード、3層の難読化）：

- **対象認証情報**: AWS IMDSv2 / GCP メタデータ / Azure / Kubernetes サービスアカウント / HashiCorp Vault / GitHub トークン / npm トークン / SSH キー / `~/.npmrc`
- **自己拡散（ワーム）**: 盗んだ npm トークンと GitHub Actions OIDC トークンを使って、被害者がwrite権限を持つ他パッケージにも悪意あるバージョンを公開
- **デッドマンスイッチ**: トークン失効を検知するとファイル削除
- **実績**: TanStack から170以上の npm パッケージ・2つの PyPI パッケージに波及

---

#### 検出と対応タイムライン

| 時刻 (UTC) | イベント |
|-----------|--------|
| 19:20 | 最初の悪意ある publish |
| 19:26 | 最後の悪意ある publish |
| 19:40〜46 | StepSecurity の ashishkurmi が GitHub Issue #7383 で報告 |
| その後 | 全影響バージョンを deprecated、npm security チームに連絡、タービス削除を依頼 |

---

#### TanStack が実施したハードニング措置

```yaml
# 修正後のワークフロー（conceptual）
on:
  pull_request_target:
    
jobs:
  benchmark-pr:
    # フォークのコードを実行しない
    # キャッシュを無効化
    # ベースリポジトリのガード追加
    if: github.event.pull_request.head.repo.owner.login == github.repository_owner
```

具体的措置：
1. `pull_request_target` ワークフローがフォークのコードを実行しないよう再構成
2. `repository_owner` ガードの追加
3. 全サードパーティ Action の参照をコミット SHA にピン留め
4. **リリースパイプラインの pnpm キャッシュを無効化**

---

#### 現在の安全性

- 現時点で公開されている **すべての TanStack パッケージの全バージョン**は安全にインストール可能
- 影響を受けたのは `@tanstack/router` と `@tanstack/start` 関連パッケージのみ（Query / DB / Store / AI / Table / Form / HotKeys / Virtual / Pacer 等は無影響）

---

### ③ なぜ重要か

1. **「信頼されたリリースパイプライン」の悪用**: 攻撃は TanStack 自身のリリースワークフローを乗っ取ったため、SLSA Provenance が有効だった。「署名済み = 安全」という前提が崩れた。
2. **キャッシュスコープの危険性**: `pull_request_target` のキャッシュがリリースワークフローと共有される点は GitHub Actions の設計上の問題であり、多くのリポジトリが同様のリスクを抱えている可能性がある。
3. **ワーム型拡散**: 単一の侵害から170以上のパッケージに拡散したことで、npm エコシステム全体の脆弱性が浮き彫りになった。
