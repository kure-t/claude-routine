## 1. TSRX — React コア OB による JSX の再構想

**記事タイトル**: TSRX — TypeScript Language Extension for Declarative UI
**URL**: [https://tsrx.dev/](https://tsrx.dev/)
**著者**: Dominic Gannaway（元 React コアエンジニア、Svelte コアメンテナー、Inferno / Lexical 作者）

### ① 目的・問題意識

JSX は React を普及させた革命的な構文拡張だったが、長年にわたり次の慢性的な問題を抱えてきた。

1. **式ベースの制約** — `if` / `for` の代わりに三項演算子と `.map()` を強制され、可読性が低下する。
2. **Hooks のルール** — Hooks は条件付きで呼び出せないため、`if` の中で `useUser()` を呼びたいだけで人工的なサブコンポーネントへの切り出しを強いられる。
3. **スタイルの分散** — CSS-in-JS 等は外部ライブラリに依存し、コンポーネント内でスタイルを共存させられない。
4. **フレームワーク依存** — JSX は React / Preact 等に結びついており、フレームワーク横断でそのまま使えない。

TSRX は TypeScript のスーパーセット（拡張子 `.tsrx`）として設計されており、宣言的 UI のための最小限の構文を追加しつつ、フレームワーク固有のセマンティクスはプラグインに委ねるアーキテクチャを採る。Ripple フレームワークから抽出され、MIT ライセンスで公開された。

### ② 具体的な技術詳細とコード例

#### コンポーネント宣言とインラインマークアップ

```tsrx
export component Greeting({ name }: { name?: string }) {
  <div class="card">
  if (name) {
    <p>"Hello, "{name}</p>
  } else {
    <p>"Hello, stranger"</p>
  }
  </div>

  <style>
  .card { padding: 1rem; }
  </style>
}
```

- `component` キーワードで関数コンポーネントを宣言する。
- テンプレート本体で `if` / `else` を式ではなく **文** として直接使える。
- `<style>` ブロックはコンポーネントにスコープされ、フレームワークプラグインが適切なスコープ付き CSS に変換する。

#### 条件分岐の中で Hooks を呼び出す

```tsrx
component Profile({ userId }: { userId: string | null }) {
  if (userId) {
    const user = useUser(userId);
    <h1>{user.name}</h1>
  } else {
    <a href="/login">"Sign in"</a>
  }
}
```

コンパイラが条件分岐を自動的にサブコンポーネントへ持ち上げ（lift）、Rules of Hooks を構造的に満たすコードを生成する。開発者は条件分岐の中で自由に Hooks を呼べる。

#### `for` ループ（インデックスとキーを明示）

```tsrx
for (const item of items; index i; key item.id) {
  <li>{`${i + 1}. ${item.text}`}</li>
}
```

- `index i` でゼロベースのインデックスを受け取る。
- `key item.id` でリスト差分キーを明示する。`.map()` と `key` プロップの組み合わせが不要になる。

#### リアクティブな props の遅延分割代入（Solid ターゲット向け）

```tsrx
component Counter(&{ count }: { count: number }) {
  <button onClick={() => count++}>
    {count}
  </button>
}
```

`&{ ... }` パターンは遅延 getter にコンパイルされ、Solid の fine-grained reactivity を維持しながら `props.count` を直接書くコードを避けられる。

#### フレームワーク対応

| ターゲット | パッケージ |
|-----------|-----------|
| React | `@tsrx/react` |
| Preact | `@tsrx/preact` |
| Ripple | `@tsrx/ripple` |
| Solid | `@tsrx/solid` |
| Vue | `@tsrx/vue` |

#### ツールチェーン

- Language Server（診断・補完・ナビゲーション）
- Prettier プラグイン
- ESLint プラグイン
- VS Code 拡張
- ソースマップサポート

### ③ なぜ効果的か

TSRX は「三項地獄」を排除し、Hooks のルール準拠のための人工的なコンポーネント分割を不要にする。スタイルをマークアップと同居させることで認知的負荷が下がる。さらにフレームワーク非依存なので、今日書いた `.tsrx` ファイルを React から Solid / Vue にコンパイルし直せる。Dominic は「エージェント時代のために設計した」と明言しており、AI が生成する UI コードが可読かつ正確であり続けるよう、制御フローを式ではなく文で表現する設計が採られている。

## 2. Understanding React Compiler — React コンパイラの仕組み解説

**記事タイトル**: Understanding React Compiler
**URL**: [https://tonyalicea.dev/blog/understanding-react-compiler/](https://tonyalicea.dev/blog/understanding-react-compiler/)
**著者**: Tony Alicea（ベストセラー開発者教育者）

### ① 目的・問題意識

`useMemo`・`useCallback`・`React.memo` による手動パフォーマンスチューニングは煩雑でエラーが起きやすく、React の再レンダリングモデルへの深い理解を必要とする。React Compiler が登場したことで開発者の混乱が生じており、本記事はコンパイラ理論の予備知識がなくても理解できるよう、React Compiler の動作原理とトレードオフを解説する。

### ② 具体的な技術詳細とコード例

#### React Compiler の正体

React Compiler は **ソースツーソース変換器（トランスパイラ）** であり、通常の JSX トランスパイルの前段として動作する。開発者が書いたコードを受け取り、自動メモ化を含む等価コードに書き換え、それを JSX トランスパイラに渡す。

#### コンパイル前後の比較

**コンパイル前（開発者が書くコード）:**

```jsx
function ExpensiveComponent({ data, multiplier }) {
  const processed = data.map(item => item.value * multiplier);

  return (
    <ul>
      {processed.map(val => <li key={val}>{val}</li>)}
    </ul>
  );
}
```

**コンパイル後（React Compiler が出力するコード、概念的な例）:**

```js
function ExpensiveComponent(t0) {
  const $ = _c(4); // キャッシュオブジェクト
  const { data, multiplier } = t0;

  let processed;
  if ($[0] !== data || $[1] !== multiplier) {
    processed = data.map(item => item.value * multiplier);
    $[0] = data;
    $[1] = multiplier;
    $[2] = processed;
  } else {
    processed = $[2];
  }

  let t1;
  if ($[3] !== processed) {
    t1 = (
      <ul>
        {processed.map(val => <li key={val}>{val}</li>)}
      </ul>
    );
    $[3] = t1;
  } else {
    t1 = $[3];
  }
  return t1;
}
```

コンパイラは `useRef` に相当するキャッシュオブジェクト `$` を挿入し、依存値が変わっていなければ再計算をスキップする。これは手動で `useMemo` を書くのと本質的に同じ処理を自動で行う。

#### メモ化のトレードオフ

- **CPU ↔ メモリのトレードオフ** — メモ化は CPU サイクルをメモリに交換する最適化。React Compiler は「できる限り多く」キャッシュに保持するため、大量データを扱うアプリではメモリ使用量の監視が必要。
- **トランスパイラとコンパイラの区別** — トランスパイラは「ソースツーソースコンパイラ」の別称であり、最適化コンパイラもコンパイラの一種。React Compiler はその両方の性質を持つ。

#### 実測パフォーマンス改善例

| 事例 | 効果 |
|------|------|
| Meta Quest Store | 初回ロード・ページ遷移が最大 12% 向上、一部操作が 2.5 倍高速化 |
| Sanity Studio | 1,411 コンポーネント中 1,231 個がコンパイル対象となり、レンダリング時間・レイテンシが全体で 20〜30% 削減 |

### ③ なぜ効果的か

開発者の多くは React Compiler を「魔法」または「単なる自動 useMemo」として曖昧に理解している。本記事は「自動メモ化にはメモリコストがある」という重要なトレードオフを明示し、コンパイラ理論の知識がない React 開発者でも読める形で正確に説明している点が価値を持つ。

## 5. ESLint Plugin React Hooks 7.1 — ESLint v10 対応

**URL**: [https://github.com/facebook/react/releases/tag/eslint-plugin-react-hooks@7.1.0](https://github.com/facebook/react/releases/tag/eslint-plugin-react-hooks@7.1.0)
**リリース日**: 2026年4月16日

### ① 目的・問題意識

ESLint v10.0.0（2026年2月7日リリース）はレガシー `.eslintrc` 設定の廃止など破壊的変更を含んでいた。`eslint-plugin-react-hooks` の旧バージョンは ESLint v10 を `peerDependencies` でサポートしておらず、CI/CD パイプラインで互換性エラーが発生していた。

### ② 具体的な技術詳細とコード例

#### ESLint v10 対応設定例

```js
// eslint.config.js (flat config)
import reactHooks from 'eslint-plugin-react-hooks';

export default [
  {
    plugins: {
      'react-hooks': reactHooks,
    },
    rules: {
      ...reactHooks.configs.recommended.rules,
    },
  },
];
```

#### v7.1 の主な変更点

- **ESLint v10 を peerDependencies に追加** — `eslint@8`・`eslint@9`・`eslint@10` すべてに対応。
- **パフォーマンス改善** — React ファイルでない場合のコンパイルをスキップし、大規模リポジトリでの lint 速度が向上。
- **Compiler Lint の改善**:
  - `set-state-in-effect` の検出精度向上
  - ref バリデーションの改善
  - より分かりやすいエラーメッセージ

### ③ なぜ効果的か

Flat Config（`eslint.config.js`）は v7.0.0 からすでにサポートされていたが、v10 の peerDependencies 非対応がブロッカーになっていた。v7.1 によりアップグレードパスが完全に整い、ESLint v10 への移行を安全に行える。

## 6. React Hook Form 7.73 — メモ化された submit と高速 deepEqual

**URL**: [https://github.com/react-hook-form/react-hook-form/releases/tag/v7.73.1](https://github.com/react-hook-form/react-hook-form/releases/tag/v7.73.1)

### ① 目的・問題意識

大規模フォームでは `handleSubmit` がレンダリングのたびに新しい関数参照を生成していたため、不要な再レンダリングを引き起こすことがあった。また、ネストの深いフィールドの型安全性と `deepEqual` の実行コストも課題だった。

### ② 具体的な技術詳細とコード例

#### メモ化された handleSubmit

```tsx
import { useForm } from 'react-hook-form';

type FormData = {
  user: {
    profile: {
      name: string;
      age: number;
    };
  };
};

function MyForm() {
  const { register, handleSubmit } = useForm<FormData>();

  // handleSubmit は v7.73 からメモ化され、不要な再レンダリングを防ぐ
  const onSubmit = handleSubmit((data) => {
    console.log(data.user.profile.name); // 深いネストも型安全
  });

  return (
    <form onSubmit={onSubmit}>
      <input {...register('user.profile.name')} />
      <input {...register('user.profile.age', { valueAsNumber: true })} />
      <button type="submit">送信</button>
    </form>
  );
}
```

#### v7.73 の主な変更点

- **memoized submit** — `handleSubmit` のメモ化により、フォームが再レンダリングされても同一の関数参照を維持。
- **高速 deepEqual** — フォームの値変化検出に使う `deepEqual` アルゴリズムを最適化し、複雑なオブジェクト比較のコストを削減。
- **深くネストしたフィールドの型安全性強化** — 複数階層にわたるフォームデータへのアクセスが TypeScript で正確に型推論される。

### ③ なぜ効果的か

`handleSubmit` のメモ化は、`React.memo` でラップした子コンポーネントへの不要な再レンダリングプロパゲーションを防ぐ。特に複雑なウィザード型フォームや、サブコンポーネントへ submit ハンドラを渡す設計で恩恵が大きい。

## 7. TanStack Store 0.11 — useAtom / useSelector / createStoreContext

**URL**: [https://github.com/TanStack/store/releases/tag/@tanstack/react-store@0.11.0](https://github.com/TanStack/store/releases/tag/@tanstack/react-store@0.11.0)

### ① 目的・問題意識

`useStore` は全 store 状態を subscibe する粗粒度の API だったため、無関係な状態変化でも再レンダリングが発生していた。また、コンテキスト経由で store を渡す標準的なパターンが存在しなかった。

### ② 具体的な技術詳細とコード例

#### useSelector（useStore の代替）

```tsx
import { Store } from '@tanstack/store';
import { useSelector } from '@tanstack/react-store';

const countStore = new Store({ count: 0, name: 'Alice' });

function Counter() {
  // count の変化時のみ再レンダリング（name の変化は無視）
  const count = useSelector(countStore, (state) => state.count);

  return (
    <button onClick={() => countStore.setState((s) => ({ ...s, count: s.count + 1 }))}>
      {count}
    </button>
  );
}
```

#### useAtom（個別アトムへのアクセス）

```tsx
import { Atom } from '@tanstack/store';
import { useAtom } from '@tanstack/react-store';

const nameAtom = new Atom('Alice');

function NameDisplay() {
  const [name, setName] = useAtom(nameAtom);

  return (
    <div>
      <p>{name}</p>
      <button onClick={() => setName('Bob')}>変更</button>
    </div>
  );
}
```

#### createStoreContext（コンテキストベースの store 共有）

```tsx
import { Store } from '@tanstack/store';
import { createStoreContext } from '@tanstack/react-store';

type AppState = { theme: 'light' | 'dark'; user: string };

const { StoreProvider, useStoreContext } = createStoreContext<Store<AppState>>();

function App() {
  const store = new Store<AppState>({ theme: 'light', user: 'Alice' });

  return (
    <StoreProvider value={store}>
      <ThemeToggle />
    </StoreProvider>
  );
}

function ThemeToggle() {
  const theme = useStoreContext((state) => state.theme);

  return <div>現在のテーマ: {theme}</div>;
}
```

#### v0.11 の変更サマリ

| 変更 | 内容 |
|------|------|
| 追加 | `useAtom` — 個別アトムへのアクセス |
| 追加 | `createStoreContext` — `StoreProvider` + `useStoreContext` |
| 変更 | `useStore` → `useSelector` に名称変更・非推奨化 |

### ③ なぜ効果的か

`useSelector` により store のうち必要な slice だけ subscribe できるため、無関係な状態更新による再レンダリングを細粒度で制御できる。`createStoreContext` は「props drilling なしに store をツリーに注入する」という React の慣用パターンを標準化するもので、Jotai の `atom` / `useAtom` に近い API を TanStack エコシステム内で利用可能にする。

## 8. Rspack 2.0 — React Server Components の実験的サポート

**URL**: [https://rspack.rs/blog/announcing-2-0](https://rspack.rs/blog/announcing-2-0)

### ① 目的・問題意識

webpack 互換の Rust 製バンドラ Rspack はビルド速度で優位性を持つが、React Server Components（RSC）のコンパイル要件（`"use client"` / `"use server"` ディレクティブの処理、クライアント/サーバー境界の検出）への対応が不十分で、Next.js 以外の RSC フレームワーク（TanStack Start 等）が Rspack を採用しにくかった。

### ② 具体的な技術詳細とコード例

#### RSC ビルド設定例

```js
// rspack.config.js
const { ReactServerComponentsPlugin } = require('@rspack/rsc-plugin');

module.exports = {
  experiments: {
    pureFunctions: true, // RSC サポートに必要
  },
  plugins: [
    new ReactServerComponentsPlugin({
      // クライアントコンパイラとサーバーコンパイラを管理するプラグイン
    }),
  ],
};
```

#### `"use client"` / `"use server"` ディレクティブの処理

```tsx
// ServerComponent.tsx（サーバーコンポーネント）
import { ClientButton } from './ClientButton'; // "use client" コンポーネントを import

export default async function ServerComponent() {
  const data = await fetchData(); // サーバーサイドでのデータ取得
  return (
    <div>
      <h1>{data.title}</h1>
      <ClientButton label="クリック" /> {/* クライアント境界を自動検出 */}
    </div>
  );
}
```

```tsx
// ClientButton.tsx（クライアントコンポーネント）
'use client';

export function ClientButton({ label }: { label: string }) {
  return <button onClick={() => alert('clicked')}>{label}</button>;
}
```

Rspack 2.0 はコンパイル時に `"use client"` / `"use server"` ディレクティブを検出し、モジュールレベルおよびファンクションレベルの `"use server"` 両方に対応する。また RSC ルール違反の React API 使用をコンパイル時にチェックし、ランタイムエラーよりも早期に問題を検出する。

#### TanStack Start との統合計画

TanStack Start の RSC 実装（[Issue #470](https://react.statuscode.com/issues/470) で紹介）が Rspack を RSC ビルドバックエンドとして採用する計画があり、webpack 互換性を維持しながら Rust ネイティブの高速ビルドで RSC を処理できるようになる。

#### パフォーマンス特性

- Rust 実装により webpack 比で数倍〜数十倍のビルド速度
- webpack の設定・ローダー・プラグインエコシステムとの互換性を維持
- `experiments.pureFunctions` フラグを有効化することで RSC サポートが利用可能（将来デフォルト化予定）

### ③ なぜ効果的か

RSC のコンパイル対応は従来 Next.js（webpack / Turbopack）に独占されていた。Rspack 2.0 により、webpack 互換エコシステムを持ちながら Rust のビルド速度を享受するプロジェクトが RSC を採用できる。TanStack Start・Remix 等の非 Next.js フレームワークでの RSC 普及を加速させる可能性がある。
