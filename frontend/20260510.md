## 1. TSRX — JSX の精神的後継：TypeScript によるUIレンダリング拡張

**記事タイトル**: TSRX — TypeScript Language Extension for Declarative UI
**発表**: [https://x.com/trueadm/status/2046696157066264846](https://x.com/trueadm/status/2046696157066264846)
**公式サイト**: [https://tsrx.dev](https://tsrx.dev)
**GitHub**: [https://github.com/Ripple-TS/ripple/tree/main/packages/tsrx](https://github.com/Ripple-TS/ripple/tree/main/packages/tsrx)

### ① 目的・問題意識

JSX は JavaScript に HTML ライクな構文を後付けしたものだが、以下の本質的な制約がある。

- **制御フローが式ベース**: `if/else` が使えず、三項演算子 `condition ? A : B` や `&&` で表現しなければならない。条件が複雑になるとネストが深くなり可読性が低下する
- **Hooks のルール制約**: `useState` などのフックは条件分岐・ループの中で呼べない（Rules of Hooks）ため、コンポーネント分割を強いられる
- **スコープ付き CSS なし**: グローバルスタイルが衝突しないよう CSS Modules や styled-components など外部の仕組みが必要
- **AI（LLM）との相性**: JSX は式の中に式が混在するため、LLM がコンテキストを把握しにくい構造になっている

TSRX（TypeScript Render Extensions）は、JSX と同じ精神で TypeScript の構文を拡張しながら、これらの問題を解決する**フレームワーク非依存のコンパイラ型 UI 言語**である。作者の **Dominic Gannaway**（@trueadm）は Inferno・Ripple の創設者であり、React コアチーム・Svelte コアチームへの参加経験を持つ。

### ② 具体的な技術詳細とコード

#### ファイル形式・インストール

TSRX のコンポーネントは `.tsrx` ファイルに記述する。フレームワークは Vite プラグイン経由で選択する。

```bash
npm install @tsrx/vite-plugin-react
```

```ts
// vite.config.ts
import { defineConfig } from 'vite';
import tsrxReact from '@tsrx/vite-plugin-react';

export default defineConfig({
  plugins: [tsrxReact()],
});
```

#### `component` キーワードによるコンポーネント定義

`function` の代わりに `component` キーワードを使う。これがコンパイラに対してレンダリングロジックの開始を示すシグナルとなる。

```tsrx
// Greeting.tsrx
export component Greeting({ name }: { name?: string }) {
  <div class="card">
    if (name) {
      <p>"Hello, "{name}</p>
    } else {
      <p>"Hello, stranger"</p>
    }
  </div>

  <style>
    .card { padding: 1rem; }
  </style>
}
```

**JSX での等価コード（比較）**:

```tsx
// Greeting.tsx (JSX)
export function Greeting({ name }: { name?: string }) {
  return (
    <div className={styles.card}>
      {name ? <p>Hello, {name}</p> : <p>Hello, stranger</p>}
    </div>
  );
}
```

TSRX では `if/else` をテンプレート内にそのまま書ける。また `class` 属性はそのまま使え、`<style>` ブロックがコンポーネントスコープの CSS として自動的に適用される（コンパイル時に CSS Modules 相当の処理が行われる）。

#### `for...of` ループによるリストレンダリング

```tsrx
// UserList.tsrx
type User = { id: number; name: string; admin: boolean; bio?: string };
type Props = { users: User[]; showBio?: boolean };

export component UserList({ users, showBio }: Props) {
  <ul class="user-list">
    for (const user of users) {
      const initials = user.name.slice(0, 2).toUpperCase();
      const role = user.admin ? 'Admin' : 'Member';

      <li class="user-row">
        <span class="avatar">{initials}</span>
        <strong>{user.name}</strong>
        <span class="badge">{role}</span>

        if (showBio && user.bio) {
          const shortBio = user.bio.slice(0, 140);
          <p class="bio">{shortBio}</p>
        }
      </li>
    }
  </ul>

  <style>
    .user-list { list-style: none; padding: 0; }
    .user-row  { display: flex; gap: 0.5rem; align-items: center; }
    .avatar    { background: #e0e0e0; border-radius: 50%; padding: 0.25rem 0.5rem; }
    .badge     { font-size: 0.75rem; color: gray; }
    .bio       { color: #666; font-size: 0.875rem; }
  </style>
}
```

`for (const item of items)` をテンプレートに直接書ける。ループ内でのローカル変数定義（`const initials = ...`）も自然なスコープで使える。

#### Hooks の位置制約を撤廃

React では Hooks を条件分岐の中に書けないが、TSRX のコンパイラが hooks の呼び出しをコンポーネント先頭に**自動的にリフト（巻き上げ）**するため、以下のように条件ブロック内で hooks を使える。

```tsrx
export component SearchResults({ query }: { query: string }) {
  if (query.length > 0) {
    // React の Rules of Hooks では本来ここに書けないが TSRX では OK
    const [page, setPage] = useState(1);
    const results = useSearch(query, page);

    for (const item of results) {
      <div>{item.title}</div>
    }
  } else {
    <p>"検索クエリを入力してください"</p>
  }
}
```

コンパイラは条件ブロックを独立したサブコンポーネントとして切り出しつつ、Hooks の制約を透過的に満たす。

#### `try / pending / catch` による非同期境界（Suspense 相当）

```tsrx
// App.tsrx
import { lazy } from 'react';
const UserProfile = lazy(() => import('./UserProfile.tsrx'));

export component App() {
  try {
    <UserProfile id={1} />
  } pending {
    <p>"Loading..."</p>
  } catch (error) {
    <p>"エラーが発生しました: "{String(error)}</p>
  }
}
```

`try` ブロックが成功時（データ取得完了）、`pending` ブロックが読み込み中、`catch` ブロックがエラー時にそれぞれ表示される。React の `<Suspense>` + `ErrorBoundary` の組み合わせを直感的な構文で統一している。

#### コンパイル後の出力（React ターゲット）

TSRX のコンパイラは `.tsrx` ファイルを標準的な React コンポーネントに変換する。

```tsx
// コンパイル後（概念的なイメージ）
import { useState, Suspense } from 'react';
import styles from './UserList.module.css'; // スコープCSS

export function UserList({ users, showBio }: Props) {
  return (
    <ul className={styles['user-list']}>
      {users.map((user) => {
        const initials = user.name.slice(0, 2).toUpperCase();
        const role = user.admin ? 'Admin' : 'Member';
        return (
          <li key={user.id} className={styles['user-row']}>
            <span className={styles.avatar}>{initials}</span>
            <strong>{user.name}</strong>
            <span className={styles.badge}>{role}</span>
            {showBio && user.bio && (
              <p className={styles.bio}>{user.bio.slice(0, 140)}</p>
            )}
          </li>
        );
      })}
    </ul>
  );
}
```

### ③ なぜ効果的か

| 特徴 | 効果 |
|------|------|
| `if/else` をテンプレートに直接記述 | 三項演算子のネスト解消・可読性向上 |
| `for...of` ループ | `Array.map()` や `key` プロパティの手動管理が不要 |
| スコープ付き CSS（`<style>` ブロック） | クラス名衝突なし・CSS Modules 設定不要 |
| Hooks の位置制約撤廃 | コンポーネントの細分化が不要になり設計がシンプルに |
| `try/pending/catch` | Suspense + ErrorBoundary を一つの構文で表現 |
| フレームワーク非依存 | React、SolidJS、Ripple を単一コードベースで対象化可能 |
| AI（LLM）との相性 | 制御フロー・スタイル・ロジックが 1 ファイルに共存し、LLM がコンテキストを把握しやすい |

TSRX は現在 **Alpha 版**として React・SolidJS・Ripple をターゲットに公開されている。`@tsrx/core` は MIT ライセンス。

## 2. React Compiler 入門 — 自動メモ化の仕組み・メリット・デメリット

**記事タイトル**: React Compiler: An Introduction, Pros, Cons & When to Use It
**URL**: [https://www.debugbear.com/blog/react-compiler](https://www.debugbear.com/blog/react-compiler)

### ① 目的・問題意識

React のパフォーマンス最適化には長らく **手動メモ化**が必要だった。

- `useMemo`: 重い計算結果をキャッシュする
- `useCallback`: 関数参照を安定化させる
- `React.memo`: コンポーネントを props 変化時のみ再レンダリングする

これらは正しく使うと効果的だが、**適用漏れ・依存配列の誤り・コードの複雑化**という問題を常に抱える。

React Compiler（以前の開発コード名: **React Forget**）は、これらのメモ化をビルド時に自動で行うコンパイラで、**2025年10月に v1.0 としてリリース**された。

### ② 具体的な技術詳細とコード

#### インストールと設定

React Compiler は **Babel プラグイン**として実装されており、Vite・Next.js・Rsbuild など主要ビルドツールと統合できる。

```bash
npm install babel-plugin-react-compiler
```

**Vite での設定例**:

```js
// vite.config.js
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';

export default defineConfig({
  plugins: [
    react({
      babel: {
        plugins: [['babel-plugin-react-compiler']],
      },
    }),
  ],
});
```

**Next.js での設定例**:

```js
// next.config.js
/** @type {import('next').NextConfig} */
const nextConfig = {
  experimental: {
    reactCompiler: true,
  },
};

module.exports = nextConfig;
```

#### 自動メモ化の仕組み（Before / After）

**Before（開発者が手動で書くコード）**:

```tsx
import { useMemo, useCallback, memo } from 'react';

const ExpensiveList = memo(({ items, onSelect }: Props) => {
  const sortedItems = useMemo(
    () => [...items].sort((a, b) => a.name.localeCompare(b.name)),
    [items]
  );

  const handleSelect = useCallback(
    (id: string) => onSelect(id),
    [onSelect]
  );

  return (
    <ul>
      {sortedItems.map((item) => (
        <li key={item.id} onClick={() => handleSelect(item.id)}>
          {item.name}
        </li>
      ))}
    </ul>
  );
});
```

**After（React Compiler を有効化した後、開発者が書くコード）**:

```tsx
// useMemo, useCallback, memo をすべて削除できる
function ExpensiveList({ items, onSelect }: Props) {
  const sortedItems = [...items].sort((a, b) => a.name.localeCompare(b.name));

  const handleSelect = (id: string) => onSelect(id);

  return (
    <ul>
      {sortedItems.map((item) => (
        <li key={item.id} onClick={() => handleSelect(item.id)}>
          {item.name}
        </li>
      ))}
    </ul>
  );
}
```

コンパイラがビルド時に依存関係を静的解析し、`useMemo` / `useCallback` / `React.memo` 相当のメモ化コードを**自動挿入**する。

#### コンパイラの動作原理

コンパイラは次の 3 段階で動作する。

1. **静的解析**: コンポーネント・フックの依存グラフを構築（どの値がどの props・state に依存するか）
2. **メモ化境界の決定**: 値・関数・JSX 要素のそれぞれについて個別にキャッシュ可能か判定
3. **コード変換**: ビルド時にキャッシュ比較コードを注入（実行時のオーバーヘッドは軽量な等値チェックのみ）

**人間が `useMemo` するよりも粒度が細かい**。`useMemo` は通常「関数全体」や「コンポーネント全体」をメモ化するが、React Compiler は JSX ノード単位・計算単位で個別にメモ化できる。

#### ESLint との連携

React Compiler は **Rules of React**（Hooks のルール・副作用の純粋性）を前提とする。違反するコードの検出には ESLint プラグインを使う。

```bash
npm install eslint-plugin-react-compiler
```

```js
// eslint.config.js
import reactCompiler from 'eslint-plugin-react-compiler';

export default [
  {
    plugins: { 'react-compiler': reactCompiler },
    rules: {
      'react-compiler/react-compiler': 'error',
    },
  },
];
```

### ③ メリット・デメリット

**メリット**

| 項目 | 説明 |
|------|------|
| コードがシンプルになる | `useMemo` / `useCallback` / `React.memo` の記述が不要 |
| 人間より細粒度のメモ化 | JSX 要素単位・個別計算単位でキャッシュ |
| 設定不要 | インストール後は自動で動作 |
| 本番実績あり | Meta の Instagram・Quest Store で効果確認（Instagram: 全ページで約 3% 改善、Quest Store: ロード 4% 以上改善） |

**デメリット・注意点**

| 項目 | 説明 |
|------|------|
| `useRef` を含む値はメモ化不可 | `useRef` の可変性・非リアクティブ性が静的解析と相容れない |
| コンポーネント間でキャッシュは共有されない | メモ化の境界はコンポーネント・フック単位 |
| Rules of React 違反で最適化スキップ | コンパイラが安全性を保証できないコードは自動でバイパス |
| エスケープハッチとして手動 memo も使える | 明示的制御が必要な箇所は引き続き `useMemo` が有効 |

## 3. React Email 6.0 — アプリに埋め込めるオープンソースのビジュアルエディタ

**記事タイトル**: React Email 6.0
**URL**: [https://resend.com/blog/react-email-6](https://resend.com/blog/react-email-6)
**エディタデモ**: [https://react.email/editor](https://react.email/editor)

### ① 目的・問題意識

React Email はコンポーネントを使ってメールテンプレートを作成するライブラリだが、従来はテンプレートのコーディングのみで、**非エンジニアが使えるビジュアルエディタ**が欠けていた。またパッケージが分散しており `@react-email/html`、`@react-email/button` など多数を個別インストールする必要があった。

React Email 6.0 の主要追加機能：
- **オープンソースのビジュアルエディタ（`@react-email/editor`）**: アプリに埋め込み可能
- **統一パッケージ（`react-email`）**: 全コンポーネントをまとめてインポート
- **カスタム拡張 API**: 独自ブロックを追加できる `EmailNode`

### ② 具体的な技術詳細とコード

#### インストール

```bash
npm install @react-email/editor
```

テーマ CSS をアプリのルートでインポートする。

```ts
// app/layout.tsx (Next.js App Router の例)
import '@react-email/editor/themes/default.css';
```

#### 基本的な埋め込み

最もシンプルな使い方（1行）。

```tsx
'use client';
import { EmailEditor } from '@react-email/editor';

const initialContent = `
  <h1>Welcome</h1>
  <p>Type "/" for a command, or select text to format it.</p>
`;

export function Editor() {
  return <EmailEditor content={initialContent} />;
}
```

`content` props は **HTML 文字列** または **TipTap JSON ドキュメント**のどちらでも受け付ける。
エディタは TipTap / ProseMirror ベースで構築されており、バブルメニュー・スラッシュコマンド・マルチカラムレイアウトが標準搭載。

#### メール HTML のエクスポート

```tsx
'use client';
import { useRef } from 'react';
import { EmailEditor, type EmailEditorRef } from '@react-email/editor';

export function MyEditor() {
  const ref = useRef<EmailEditorRef>(null);

  const handleExport = async () => {
    if (!ref.current) return;
    const html = await ref.current.getEmailHTML();
    console.log(html); // メールクライアント互換の HTML
  };

  return (
    <div>
      <EmailEditor ref={ref} content="<p>Edit me</p>" />
      <button onClick={handleExport}>Export HTML</button>
    </div>
  );
}
```

`getEmailHTML()` は各メールクライアント（Gmail・Outlook・Apple Mail 等）で正しくレンダリングされる HTML を返す。

#### カスタム拡張ブロック（`EmailNode`）

独自のブロックを追加するための `EmailNode` API を使ったカスタム拡張の例。

```tsx
import { EmailNode } from '@react-email/editor';
import type { Node } from '@tiptap/core';

export const BannerNode: EmailNode = {
  name: 'banner',
  group: 'block',
  // エディタ上での React 表示
  renderReact: ({ node }) => (
    <div style={{ background: '#fef3c7', padding: '1rem', borderRadius: '0.5rem' }}>
      {node.attrs.text}
    </div>
  ),
  // メール送信用 HTML 出力
  renderToHtml: ({ node }) =>
    `<table width="100%"><tr><td style="background:#fef3c7;padding:16px">${node.attrs.text}</td></tr></table>`,
  addAttributes: () => ({
    text: { default: 'バナーテキスト' },
  }),
};
```

CDN 画像アップロード・インラインチャートなども同じ API で追加できる。

#### 統一パッケージ

```tsx
// Before（6.0 以前）
import { Html }    from '@react-email/html';
import { Button }  from '@react-email/button';
import { Section } from '@react-email/section';

// After（6.0 以降）
import { Html, Button, Section } from 'react-email';
```

### ③ なぜ効果的か

- **ノーコードでメール作成**: デザイナーや非エンジニアがアプリ内で直接メールを編集できる
- **全メールクライアント対応 HTML**: TipTap ベースのエディタが各クライアントの quirk を処理した HTML を出力
- **1行で埋め込み可能**: ホスティング・独自ツール構築なしに自社アプリへ統合できる
- **カスタム拡張が容易**: `EmailNode` API でビジネス固有のブロックを追加できる
- **オープンソース**: MIT ライセンス・196 名以上のコントリビュータ

## 4. Expo が $45M のシリーズ B を調達、Expo Agent を発表

**発表日**: 2026年4月16日

### ① 目的・問題意識

React Native の主要プラットフォームである Expo は、モバイルアプリ開発のツールチェーン（EAS Build・EAS Submit・Expo Router 等）を提供している。課題は「アイデアからプロダクション品質のアプリまでの距離が長く、専門知識を持つエンジニアのサポートが企業内に必要」という点。今回の調達は、**Expo Agent**という AI 駆動のモバイル開発支援機能の提供と技術拡張のために行われた。

### ② 具体的な技術詳細

#### 調達内容

| 項目 | 内容 |
|------|------|
| 調達額 | $45M（シリーズ B） |
| リード投資家 | Georgian |
| 参加投資家 | Leadout Capital、A.Capital Ventures、Red Swan Ventures |

#### Expo Agent

**Expo Agent** は、エンタープライズチームに向けた「常駐モバイル専門家」の役割を担う AI ツール。

- アプリ開発の技術的な質問に回答するエキスパートAI
- プロジェクト設定・ビルドエラー・デプロイの問題を解決
- 企業にとってモバイル専任エンジニアを確保するコストを削減

#### 組織面

- Seth Webster が **Chief Developer Evangelist** として参画
