## 1. Googleによる「バックボタンハイジャック」新スパムポリシー

**参照元:** [Introducing a new spam policy for "back button hijacking" — Google Search Central Blog](https://developers.google.com/search/blog/2026/04/back-button-hijacking)

### ① 目的・問題意識

「バックボタンハイジャック（Back Button Hijacking）」とは、ウェブサイトがブラウザのHistory APIを悪用し、ユーザーが「戻る」ボタンを押しても直前の外部ページに戻れないよう妨害する手法のこと。ユーザーが想定しないページに誘導したり、広告・オファーのオーバーレイを表示したりするために使われる。

Googleは2026年6月15日より、この行為をスパムポリシー違反として認定し、検索ランキングの自動降格またはManual Spam Actionの対象とすることを発表した（告知日: 2026年4月13日）。Googleによる指導内容:

> "Ensure you are not doing anything to interfere with a user's ability to navigate their browser history."  
> — Chris Nelson (Google)

### ② 具体的な技術詳細とコード

#### ハイジャックの典型的な実装パターン

```javascript
// 【悪用例】バックボタンハイジャックのパターン
// 1. ページ読み込み時に「偽のエントリ」を履歴に追加する
history.pushState(null, '', window.location.href);

// 2. popstate イベントを監視してユーザーの「戻る」操作を横取りする
window.addEventListener('popstate', () => {
  // ユーザーをサイト内に留める処理（割引表示、別ページへのリダイレクトなど）
  showExitConfirmation();
  // 再度偽エントリを追加してトラップを維持
  history.pushState(null, '', window.location.href);
});
```

**仕組みの説明:**
- `history.pushState()` を呼び出すとブラウザ履歴スタックに偽エントリが積まれる
- 例: `[前ページ]` → `[当サイト]` → `[当サイト（偽エントリ）]`
- ユーザーが「戻る」を押すと偽エントリに移動し、`popstate` イベントが発火。サイトは別ページへのリダイレクトや広告表示などを実行する

#### ハイジャックの検出方法

1. **手動確認**: 「戻る」ボタンを右クリックして履歴プレビューを確認。同一ページが複数回並んでいる場合は要注意
2. **Chrome DevTools**: Sources → Event Listeners → `popstate` でリスナーの有無を確認
3. **コードレビュー**: `history.pushState()`、`history.replaceState()`、`popstate` イベントリスナーを検索して意図しない呼び出しがないか確認

#### 修正方法

```javascript
// ✅ 正しい使い方: 実際のルート変更時のみ pushState を呼ぶ
function navigateTo(path) {
  history.pushState({ page: path }, '', path);
  renderPage(path);
}

// ❌ 避けるべきパターン: UI トグルなどで pushState を呼ぶこと
function toggleModal() {
  // これは履歴を汚染する
  // history.pushState(null, '', window.location.href);
  modal.classList.toggle('open');
}
```

**SPAフレームワーク（React / Vue / Angular）での対策:**
- ルーター設定を見直し、実際のページ遷移のみ`pushState`が呼ばれるようにする
- 不要な再レンダリングやデータフェッチで`pushState`を呼んでいないか確認
- サードパーティ広告スクリプトやライブラリのHistory API操作も確認対象

### ③ なぜそれが効果的か

Googleは**2ヶ月前の告知期間（2026年4月〜6月15日）**を設けており、サイト運営者が修正する余裕を与えている。施行後は**ランキング降格や手動スパム対応**が発生し得る。特にSPAやJavaScript重視のサイト、広告プラットフォームを使用するサイトで見落とされがちな問題であり、6月15日までの対応が必須。

## 2. CSS Name-Only Container Queries — サイズ条件なしでスコープを実現

**参照元:** [Name-only @container queries: A solution to the naming wars — WebKit Blog](https://webkit.org/blog/17923/name-only-container-queries-a-solution-to-the-naming-wars/) (Published: April 13, 2026)

Also: [Name-Only Containers: The Scoping We Needed — Frontend Masters Blog](https://frontendmasters.com/blog/name-only-containers-the-scoping-we-needed/) (Published: April 7, 2026)

### ① 目的・問題意識

従来のContainer Queryは**コンテナの幅（サイズ）による条件**が必須だった。コンポーネントベースの開発において、グローバルなCSSのカスケード汚染を防ぐためには「スコープ（スタイルの局所化）」が必要だが、CSS Modulesのようなビルドツールや、BEM/SUIT等の命名規則に頼る必要があった。

**Name-Only Container Queries**（Safari 26.4 で実装）は、サイズ条件なしでコンテナ名だけを使ってスタイルをスコープできる新機能。

### ② 具体的なコードと技術詳細

#### 従来のContainer Query（サイズ条件が必要）

```css
/* 従来: サイズ条件（min-width など）が必須 */
.sidebar {
  container-name: sidebar;
  container-type: inline-size;
}

@container sidebar (min-width: 400px) {
  .card {
    padding: 2rem;
  }
}
```

#### Name-Only Container Query（サイズ条件なし）

```css
/* ✅ 新機能: サイズ条件なしでコンテナ名だけを使う */
.sidebar {
  container-name: sidebar;
  /* container-type を省略可能 */
}

@container sidebar {
  .card {
    padding: 1rem;
    border-radius: 8px;
  }
  .card-title {
    font-size: 1.2rem;
  }
}
```

#### コンポーネントスコープの実用例

```css
/* ds-card コンポーネントのスタイル */
.ds-card {
  container-name: ds-card;
}

@container ds-card {
  /* ファイル内の .title は ds-card スコープ内のみに適用 */
  .title {
    font-weight: bold;
    color: var(--card-title-color);
  }
}

/* ds-article コンポーネントのスタイル */
.ds-article {
  container-name: ds-article;
}

@container ds-article {
  /* .title はここでは ds-article スコープ */
  .title {
    font-size: 2rem;
    color: var(--article-title-color);
  }
}
```

両者の `.title` クラスは衝突しない。フォルダ構造の一意性をCSS自体に持ち込むことができる。

**@scope との比較:**

| 特徴 | Name-Only @container | @scope |
|---|---|---|
| サイズ条件 | 不要 | 不要 |
| "ドーナツスコープ" | なし | あり（子孫の範囲制限可） |
| ユースケース | コンポーネント単位の名前空間 | より細かいカスケード制御 |

#### ブラウザサポート（2026年4月時点）

- **Safari 26.4**: サポート済み
- **Chrome 148**: サポート予定（間もなく）
- **Firefox**: 未サポート

### ③ なぜそれが効果的か

CSS Modulesや命名規則（BEM）なしに**純粋なCSSだけでコンポーネントのスタイルをスコープ**できる。ビルドツールへの依存を減らし、コンポーネント内の `.title` や `.button` といった汎用クラス名を安全に使い回すことが可能になる。ネーミング論争（命名規則戦争）をなくす解決策として注目されている。

## 3. ブレークポイントなしのUIを構築する — 現代CSS的レスポンシブ設計

**参照元:** [Building a UI Without Breakpoints — Frontend Masters Blog](https://frontendmasters.com/blog/building-a-ui-without-breakpoints/) (Published: April 15, 2026)

### ① 目的・問題意識

従来のメディアクエリ（ブレークポイント）によるレスポンシブデザインは、ビューポート幅を基準にしていた。しかし現代のUIは**コンポーネント単位**で構築されており、同じコンポーネントがサイドバーにも、メインコンテンツ内にも使われる。その場合、「ビューポート幅」は適切な基準ではなく、コンポーネントが実際に配置されている**コンテキストに応じたサイズ**が重要となる。

この記事は「ブレークポイントをなくす」のではなく、**最後の手段として限定的に使い、基本はCSS本来の intrinsic layout で適応させる**設計思想を提案する。

### ② 具体的なコードと技術詳細

#### コンテナユニット（cqi）を使ったスケーリング

```css
/* コンテナを宣言 */
.card-container {
  container-type: inline-size;
  container-name: card;
}

/* コンテナ幅に対してフォントサイズをスケール */
@container card {
  .card-title {
    /* cqi = コンテナの inline サイズに対する % */
    font-size: clamp(1rem, 3cqi, 2.5rem);
  }
}
```

#### `clamp()` による fluid border-radius

```css
/* ビューポートではなくコンテナサイズに応じて角丸が変化 */
.card {
  container-type: inline-size;
  border-radius: clamp(4px, calc(-16.87px + 6.522cqi), 64px);
}
```

`clamp(最小値, 流動値, 最大値)` パターンにより、コンテナが小さい時は `4px`、大きくなるにつれて `64px` まで滑らかに変化。

#### Intrinsic レイアウト（幅条件なし）

```css
/* flex: calc(infinity) 1 40cqi
   - grow: 無限（利用可能な領域を全て使う）
   - shrink: 1（縮小可）
   - basis: 40cqi（コンテナ幅の40%が基準） */
.card-content {
  flex: calc(infinity) 1 40cqi;
  min-width: 0;
}
```

これにより「40cqi 以下には縮まない」という制約を持ちつつ、利用可能なスペースを動的に占有する。

#### 全体的なアーキテクチャ

```css
/* ① Intrinsic Layout: flexbox/grid が構造を担う */
.card-grid {
  display: grid;
  grid-template-columns: repeat(auto-fill, minmax(min(280px, 100%), 1fr));
}

/* ② Fluid Values: clamp() がスケールを担う */
.heading {
  font-size: clamp(1.25rem, 2.5cqi, 2rem);
  padding: clamp(0.5rem, 2cqi, 1.5rem);
}

/* ③ Container Units: コンテキスト適応を担う */
.sidebar .widget {
  container-type: inline-size;
  font-size: clamp(0.8rem, 4cqi, 1rem);
}

/* ④ Container Queries: 明示的な構造変化のためだけ使う */
@container (min-width: 400px) {
  .card {
    flex-direction: row;
  }
}

/* ⑤ Media Queries: デバイス能力やユーザー設定のためだけ使う */
@media (prefers-reduced-motion: reduce) {
  .transition { animation: none; }
}
@media print {
  .no-print { display: none; }
}
```

### ③ なぜそれが効果的か

従来のブレークポイントは「320px, 768px, 1024px」のような**絶対的なビューポート幅**に依存するため:
- コンポーネントが配置されるコンテキストを無視している
- ページに追加するたびにブレークポイントを調整し直す必要がある
- レイアウト変更のたびに大量のCSSが変化する

**この手法が優れている点:**
1. **コンポーネントが自律的**に振る舞う — 配置場所に関わらず適切にレンダリングされる
2. **CSSの量が減り**、バグも減る — 条件分岐が少なくなる
3. **ブレークポイントは意図的な例外**として使う — 「32px以下では2カラムに」など明確な目的を持ったルールのみ

> "When intrinsic layout, fluid values, container units, and container queries become your default toolbox, breakpoints stop being the thing you depend on first — they become a precise tool you use on purpose."

## 関連: Chrome 147 の新機能（2026年4月7日リリース）

*以下は同時期（April 8号 Issue #736 前後）にカバーされた Chrome 147 の主要機能。*

### CSS `contrast-color()` 関数 — Baseline Newly Available

```css
/* テキストカラーを自動で黒or白に切り替える */
.card {
  background-color: var(--user-color);
  /* contrast-color() が背景に対してより高コントラストな色（black/white）を返す */
  color: contrast-color(var(--user-color));
}

/* 直接使用例 */
.badge {
  background: #ff6b6b;
  color: contrast-color(#ff6b6b); /* → black が返る */
}

/* feature detection */
@supports not (color: contrast-color(red)) {
  .badge {
    color: #000; /* fallback */
  }
}
```

- Chrome 147, Firefox, Safari で Baseline Newly Available に
- `black` または `white` のみを返す（現仕様の制限）
- WCAG AA コントラスト要件を自動的に満たすためのアクセシビリティツール

### Element-scoped View Transitions — `element.startViewTransition()`

```javascript
// Chrome 147以前: ドキュメント全体のView Transition
document.startViewTransition(() => {
  updateDOM();
});

// Chrome 147以降: 特定要素にスコープされたView Transition
const list = document.querySelector('ul');
list.startViewTransition({
  callback: () => {
    // <ul> 内のDOM操作のみ
    list.innerHTML = newContent;
  },
});
```

```css
/* スコープ設定 */
#my-scope {
  contain: layout;
  view-transition-scope: all;
}

#participant {
  view-transition-name: card;
}
```

**利点:**
- 複数要素で同時に独立したView Transitionを実行できる
- スコープ外のページ部分はインタラクティブなまま（応答停止なし）
- 入れ子のView Transitionに対応
- `position: fixed` のレイヤリング問題が解決

### `Math.sumPrecise()` — 精度を保った数値の合計

```javascript
// 従来の問題: 浮動小数点精度の損失
let sum = 0;
for (const n of [1e20, 0.1, -1e20]) {
  sum += n;
}
console.log(sum); // → 0（誤り: 中間値 1e20+0.1 が 1e20 に丸められる）

// Math.sumPrecise() による解決
console.log(Math.sumPrecise([1e20, 0.1, -1e20])); // → 0.1（正確）

// 基本的な使用例
console.log(Math.sumPrecise([1, 2, 3]));    // → 6
console.log(Math.sumPrecise([]));            // → -0（空の場合は -0 を返す）

// 注意: 浮動小数点表現自体の限界は超えられない
console.log(Math.sumPrecise([0.1, 0.2]));   // → 0.30000000000000004
// （0.1 と 0.2 自体がバイナリ浮動小数点で正確に表現できないため）
```

- Chrome 147 で Baseline Newly Available
- 統計計算、金融計算など精度が重要な場面で有効
- イテラブル全般を受け取れる（Arrayだけでなく、SetやMapなども可）
