## 1. Google の新スパムポリシー「バックボタン・ハイジャッキング」

**記事タイトル**: Introducing a new spam policy for "back button hijacking"
**URL**: [https://developers.google.com/search/blog/2026/04/back-button-hijacking](https://developers.google.com/search/blog/2026/04/back-button-hijacking)
**著者**: Chris Nelson（Google）
**発表日**: 2026年4月13日
**執行開始**: 2026年6月15日

### ① 目的・問題意識

バックボタン・ハイジャッキング（Back Button Hijacking）とは、Webサイトがブラウザのバック機能に干渉し、ユーザーが以前訪れたページに戻れないようにする欺瞞的な手法。具体的には次のような状況を引き起こす。

- ユーザーが「戻る」を押すと、訪問したことのないページにリダイレクトされる
- 同サイト内の別ページ（広告・推薦コンテンツなど）にループさせられる
- 複数回「戻る」を押さないと元のサイトに戻れない
- 離脱を妨げるインタースティシャル広告や「出口ポップアップ」が表示される

Googleは同手法を **「悪意のある慣行」スパムポリシーの明示的な違反** として位置付け、2026年6月15日以降に対象ページへのランキング降格やスパムアクションを適用すると発表した。

> "Ensure you are not doing anything to interfere with a user's ability to navigate their browser history."
> — Chris Nelson, Google

### ② 具体的な技術詳細

#### バックボタン・ハイジャッキングの仕組み

主に **History API**（`history.pushState()` / `history.replaceState()`）を悪用して実装される。本来 SPA（シングルページアプリ）がURL変更をフルページリロードなしに行うための正規APIだが、ハイジャッカーはこれを「ファントム（幽霊）エントリー」の注入に使用する。

```javascript
// ❌ バックボタン・ハイジャッキングの典型的なパターン
// ページロード直後にフェイクのヒストリーエントリーを大量注入
(function trapUser() {
  for (let i = 0; i < 5; i++) {
    // 実際のナビゲーションを伴わない"幽霊エントリー"を追加
    history.pushState({ trapped: true, step: i }, '', window.location.href);
  }

  // バックボタンを検知してサイト内にリダイレクト
  window.addEventListener('popstate', function(event) {
    if (event.state && event.state.trapped) {
      // 別の広告ページやコンテンツへ強制移動
      window.location.href = '/monetized-interstitial?ref=back';
    }
  });
})();
```

```javascript
// ❌ replaceState を使った履歴の書き換えパターン
window.addEventListener('load', function() {
  // 現在のエントリーを別のURLに置き換える
  history.replaceState(
    { hijacked: true },
    '',
    '/fake-referral-page'
  );
  history.pushState(
    { actual: true },
    '',
    window.location.href
  );
});

// ユーザーが「戻る」を押すと /fake-referral-page に遷移させられる
```

```javascript
// ❌ popstate をインターセプトして離脱をブロックするパターン
window.addEventListener('popstate', function(e) {
  e.preventDefault(); // 実は preventDefault は popstate では効かないが、
                       // 即座に pushState を呼んで"元に戻す"手法
  history.pushState(null, '', window.location.href);
  showExitIntentOverlay(); // 離脱阻止ポップアップを表示
});
```

#### 正当な使用との違い

```javascript
// ✅ 正当な History API の使用例（SPA のルーティング）
// React Router や Next.js のような SPA フレームワーク内でのナビゲーション
function navigateTo(path) {
  // ユーザーが実際にページ遷移を意図したときのみ pushState を呼ぶ
  history.pushState({ path }, '', path);
  renderPage(path); // 対応するコンテンツを実際にレンダリング
}

// ✅ フィルター・ソートの URL 反映（状態の保存）
function updateFilters(filters) {
  const params = new URLSearchParams(filters);
  // ユーザーの明示的な操作（フィルター変更）に基づく更新
  history.pushState({ filters }, '', `?${params}`);
}

// ❌ NG: コンポーネントの再レンダリングや
//        データフェッチのたびに pushState を呼ぶのは違反になりうる
```

#### 問題のある実装パターン一覧

| パターン | 技術的手法 | 違反に該当するか |
|---------|-----------|----------------|
| ページロード時に複数の偽エントリーを注入 | `history.pushState()` ループ | **違反** |
| バックボタンを検知して別ページへリダイレクト | `popstate` + `location.href` 変更 | **違反** |
| 現在の URL を離脱時に別 URL に置き換え | `history.replaceState()` の悪用 | **違反** |
| 広告スクリプトがヒストリーを操作 | サードパーティスクリプト | **違反（責任はサイトオーナー）** |
| SPA ルーターによる正当な URL 管理 | フレームワークのナビゲーション | 問題なし |
| ページフィルターのURL反映 | `replaceState` による状態保存 | 問題なし |

#### 検出方法（Chrome DevTools）

```javascript
// 1. Chrome DevTools を開く（F12）
// 2. Application タブ → Session Storage を確認
// 3. コンソールで以下を実行して history 操作を監視:

const originalPushState = history.pushState;
history.pushState = function(...args) {
  console.trace('pushState called:', args);
  return originalPushState.apply(this, args);
};

const originalReplaceState = history.replaceState;
history.replaceState = function(...args) {
  console.trace('replaceState called:', args);
  return originalReplaceState.apply(this, args);
};

// バックボタン押下後、単一クリックで前のドメインに戻れるか確認
```

#### 対処・修正方法

```javascript
// ① SPA フレームワーク（React Router）の設定を見直す
// router.push() がユーザーの明示的なアクション以外で呼ばれていないか確認

// ❌ useEffect 内で意図せず呼ばれる例
useEffect(() => {
  router.push('/some-path'); // データ取得のたびに呼ばれる → 違反の可能性
}, [data]);

// ✅ ユーザーインタラクションに基づく呼び出し
// <button onClick={() => router.push('/next-step')}>次へ</button>
```

```bash
# ② サードパーティスクリプトの監査
# Chrome DevTools のネットワークタブでサードパーティスクリプトをブロックし、
# バックボタンの動作が正常になるか確認
# 広告プラットフォームやアナリティクスのスクリプトも対象
```

```
③ セルフテスト：1クリックで前のドメインに戻れるか確認
  サイト訪問後、バックボタンを1回クリックして：
  - 検索結果ページ（Google/Bing など）に戻れる → 問題なし
  - 同じページや別の自サイトページが表示される → 要修正
```

### ③ なぜ効果的か（Googleのポリシーが重要な理由）

| 観点 | 内容 |
|------|------|
| **ユーザー体験の保護** | ブラウザの「戻る」は最も基本的なナビゲーション手段。これを妨げることはユーザーの信頼を根本的に損なう |
| **ペナルティの強さ** | 手動スパムアクション（Search Console で通知）または自動ランキング降格。サイト全体に影響する可能性 |
| **サードパーティ責任** | 広告プラットフォームやライブラリによる実装でも、**サイトオーナーが責任を負う** |
| **6月15日の期限** | 2ヶ月前の事前通知あり。早急な技術監査が必要 |
| **SPAへの影響** | React / Vue / Angular 等のフレームワークを使用するサイトは特に注意が必要 |

## 2. 補足: Chrome 147 の主要新機能（同週リリース）

**リリース日**: 2026年4月7日
**参考**: [Chrome 147 Release Notes](https://developer.chrome.com/release-notes/147) / [New in Chrome 147](https://developer.chrome.com/blog/new-in-chrome-147)

> Issue 737 発行（4/15）の直前にリリースされた Chrome 147 の主要新機能。

### ① CSS `contrast-color()` 関数（Baseline 新規対応）

#### 目的・問題意識

テキストのコントラスト比をアクセシビリティ基準（WCAG）に合わせて管理するのは複雑だった。背景色を動的に変えると、テキスト色も手動で変更する必要があった。

#### 具体的なコード

```css
/* contrast-color() は背景色に対して最大コントラストを提供する
   black か white を自動で返す */

.badge {
  background-color: var(--user-color);
  /* 背景色に応じて自動的に黒か白を選択 */
  color: contrast-color(var(--user-color));
}

/* 実用例：動的テーマカラーに対応したボタン */
:root {
  --brand-color: oklch(60% 0.15 240);
}

.button {
  background: var(--brand-color);
  /* oklch(60% 0.15 240) に対して読みやすい方の色を自動選択 */
  color: contrast-color(var(--brand-color));
  padding: 0.5em 1em;
  border: none;
}
```

#### なぜ効果的か

- アクセシビリティ対応のコードを大幅に削減
- JavaScript で色を計算する必要がなくなる
- 全主要ブラウザで Baseline 対応（Chrome 147, Firefox 150, Safari 26.5）

### ② CSS `border-shape` プロパティ

#### 目的・問題意識

非矩形のボーダーを作るには `clip-path` や複雑な CSS ハックが必要だった。

#### 具体的なコード

```css
/* border-shape で任意の形状のボーダーを定義 */
.card {
  border: 4px solid oklch(50% 0.2 260);
  border-shape: round(50%);
}

.dialog {
  border: 2px solid navy;
  border-shape: polygon(
    0 8px, 8px 0, calc(100% - 8px) 0,
    100% 8px, 100% calc(100% - 8px),
    calc(100% - 8px) 100%, 8px 100%,
    0 calc(100% - 8px)
  );
}
```

#### なぜ効果的か

- デザインの自由度が大幅に向上
- `clip-path` のような要素の外側を切り取るアプローチ���異なり、ボーダーのみを形状変更
- カード、バッジ、ダイアログなどに独自の形状を持たせられる

### ③ 要素スコープの View Transitions（`Element.startViewTransition()`）

#### 目的・問題意識

従来の View Transitions API は `document.startViewTransition()` でページ全体にトランジションを適用してい��。特定のコンポーネント内でのみトランジションを行いたい場合、影響範囲の制御が難しかった。

#### 具体的なコード

```javascript
// 特定の要素スコープでトランジション
const container = document.getElementById('my-widget');

// document.startViewTransition() ではなく element.startViewTransition() を使用
const transition = container.startViewTransition(() => {
  // この更新のみがトランジション対象になる
  container.innerHTML = newContent;
});

// 複数の要素で同時にトランジションを実行できる
const header = document.querySelector('header');
const sidebar = document.querySelector('.sidebar');

// これらは独立したトランジションとして同時実行可能
header.startViewTransition(() => updateHeader());
sidebar.startViewTransition(() => updateSidebar());
```

```css
/* 要素スコープのトランジション用 CSS */
#my-widget::view-transition-old(root) {
  animation: slide-out 0.3s ease-out;
}

#my-widget::view-transition-new(root) {
  animation: slide-in 0.3s ease-in;
}

@keyframes slide-out {
  to { transform: translateX(-100%); }
}

@keyframes slide-in {
  from { transform: translateX(100%); }
}
```

#### なぜ効果的か

- トランジションの影響範囲をコンポーネント単位に限定できる
- 複数のトランジションを同時並行で実行可能
- 祖先要素の clip や transform の影響を受けるため、レイアウトとの整合性が保たれる

---

### ④ `<video>` / `<audio>` の遅延読み込み（Lazy Loading）

#### 目的・問題意識

`<img>` と `<iframe>` は `loading="lazy"` で遅延読み込みができたが、`<video>` と `<audio>` には対応していなかった。

#### 具体的なコード

```html
<!-- Chrome 147 から video と audio も loading="lazy" に対応 -->
<video
  src="large-video.mp4"
  loading="lazy"
  controls
  width="800"
  height="450"
>
  <!-- ビューポートに近づくまで読み込まれない -->
</video>

<audio
  src="podcast-episode.mp3"
  loading="lazy"
  controls
>
</audio>

<!-- 複数のソースを持つ場合 -->
<video loading="lazy" controls>
  <source src="video.webm" type="video/webm">
  <source src="video.mp4" type="video/mp4">
</video>
```

#### なぜ効果的か

- ページ初期ロード時のネットワーク使用量を削減
- 特にメディアコンテンツが多いページで LCP（Largest Contentful Paint）の改善に貢献
- `loading="lazy"` の一貫したインターフェースで統一的に管理可能

## 3. 補足: Web プラットフォームの 2026年4月の新着情報

#### `Math.sumPrecise`（JavaScript）

```javascript
// 従来の浮動小数点誤差の問題
console.log(0.1 + 0.2); // 0.30000000000000004 ← 誤差あり

// Math.sumPrecise は正確な合計を返す
const values = [0.1, 0.2, 0.3];
console.log(Math.sumPrecise(values)); // 0.6 ← 正確

// 大量の数値の合計にも有効
const prices = [9.99, 14.99, 4.99, 19.99];
const total = Math.sumPrecise(prices); // 49.96 ← 正確
```

#### `<img sizes="auto">` — レスポンシブ画像の自動最適化

```html
<!-- Firefox 150: sizes="auto" で lazy-loaded 画像が
     実際のレイアウトサイズから最適な srcset を自動選択 -->
<img
  srcset="
    small.jpg  400w,
    medium.jpg 800w,
    large.jpg  1200w
  "
  sizes="auto"
  loading="lazy"
  alt="Responsive image"
>
<!-- 従来は sizes="(max-width: 600px) 100vw, 800px" のように手動指定が必要だった -->
```
